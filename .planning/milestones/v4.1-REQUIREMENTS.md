# Requirements: v4.1 Feedback-Driven Polish

**Defined:** 2026-04-19
**Core Value:** Close the gap between "the app has features" and "the app is daily-useful." First-week feedback surfaced correctness bugs, UX friction, and a structural gap (per-partner customization) that blocks several downstream features.

## Theme

The polish + correctness pass that makes v4.0 actually usable day-to-day, plus the single foundational feature (Partner Config) that unlocks product-type pre-population, sub-cohort analysis (Snap EN/ES, Happy Money banks), and segment-aware projections in v5.0.

## KPI Improvements (Phase 38)

- [x] **KPI-01**: KPI cards render a graduated cascade based on the cohort's max batch age — the app shows something meaningful from day one rather than gating on 12 months of data. Rules:
  - **< 3 months**: one card — "Rate since inception" (collection rate on everything placed so far, regardless of age)
  - **3 to < 6 months**: two cards — "3mo rate" + "Rate since inception"
  - **6 to < 12 months**: two cards — "3mo rate" + "6mo rate"
  - **≥ 12 months**: two cards — "6mo rate" + "12mo rate"

  The cascade is driven by max batch age in the current scope (partner + product + filters), not a global setting. Cards that don't apply at the current age are not rendered — no "insufficient data" placeholder tiles.
- [x] **KPI-02**: KPI delta labels explicitly state the comparison baseline (e.g., "vs 3-batch rolling avg" rather than an unlabeled percentage)
- [ ] **KPI-03** *(Deferred to Phase 41 — requires new Snowflake column `ACCOUNTS_WITH_PLANS_AT_{6,12}_MONTH`; `TOTAL_ACCOUNTS_WITH_PLANS` today is lifetime. See `.planning/phases/38-polish-correctness-pass/38-CONTEXT.md`.)*: 6mo and 12mo rate cards display the matching-horizon commitment rate as a secondary line (shown only when the corresponding rate card is shown per KPI-01's cascade)
- [x] **KPI-04**: Rolling-average deltas are suppressed when the cohort is too young to support a meaningful rolling avg at the card's horizon (e.g., a 3-batch rolling average of 6mo rate requires at least 3 prior batches that have reached 6 months of age). When suppressed, the card shows the rate value with no delta indicator — no misleading "vs N/A" or "0.0%" labels. Once Phase 40 (Projected Curves v1) ships, "vs projected curve" becomes the natural fallback comparison for early batches (PRJ-04).

## Metabase Import Polish (Phase 38)

**Context**: Phase 37 shipped a heuristic chart-type inference inside `chart-inference.ts` — the parser guesses (GROUP BY + aggregate → bar, time-series X + numeric Y → line, else no chart). SQL alone cannot carry Metabase's visualization setting because Metabase stores `display` + `visualization_settings` in a separate card JSON, not in the SQL string. First-week feedback: users expected chart type to round-trip from their Metabase card; the heuristic is right often but not always, and there's no user recourse when it's wrong.

- [x] **MBI-01**: The Metabase Import Preview step exposes a chart-type override control in the Chart section. The heuristic-inferred type pre-selects; the user can pick line / scatter / bar / none from a dropdown (mirroring `ChartBuilderToolbar`'s chart-type segmented control from Phase 36 so the affordance is consistent) before clicking Apply. The override also lets the user pick X/Y axes when the heuristic leaves them ambiguous. The Chart section shows the inference reason as helper text ("Inferred from GROUP BY + AVG aggregate") so users know why the default was chosen.

## Partner Config (Phase 39 — foundational feature)

**Principle**: Blending metrics across a partner's product lines (e.g., Happy Money 1st-party + 3rd-party in a single view) produces misleading numbers because the economics, recovery curves, and rates are structurally different. The app must enforce per-product scoping as the default — product-splitting is a correctness fix, not a user preference. Segment-splitting (Snap EN/ES, Happy Money banks within a single product) is the user-configurable layer on top.

- [x] **PCFG-01**: Product type is derived from Snowflake's `ACCOUNT_TYPE` column — no config record, no editable field. The canonical unit of analysis across the app is the `(PARTNER_NAME, ACCOUNT_TYPE)` pair, not `PARTNER_NAME` alone. — *Plan 39-01: pair encoding helpers in `src/lib/partner-config/pair.ts` derive product from ACCOUNT_TYPE at read time across SidebarDataPopulator, buildPairSummaryRows, and groupByPair. No storage module touched.*
- [x] **PCFG-02**: The sidebar partner list shows one row per `(partner, product)` pair derived from `SELECT DISTINCT PARTNER_NAME, ACCOUNT_TYPE`. Single-product partners look identical to today; multi-product partners split into N rows (e.g., "Happy Money — 1st Party", "Happy Money — 3rd Party"). — *Plan 39-01: SidebarDataPopulator emits one row per pairKey; displayNameForPair suffixes when productsPerPartner > 1.*
- [x] **PCFG-03**: Partner selection state carries both `partner` and `product`. All downstream scoping — KPIs, charts, table, drill-downs, filters, Partner Lists — keys off the pair. Nothing in the app aggregates across products for a single partner. — *Plan 39-01: DrillState.product + URL `?pr=`; usePartnerStats / partnerRows / batchRows / tableData / rootSummaryRows / cross-partner ranking / anomalies / AI context all key off pair.*
- [x] **PCFG-04**: No UI path produces a cross-product blended view for a single partner. Blending is blocked at the selection layer; it is not a user-controllable toggle. (Cross-*partner* aggregation is unchanged — different partners remain independent, blendable entities.) — *Plan 39-01: filter popover dropped partner combobox (Open Q #2 deprecation); sidebar has no partner-level click target for multi-product partners; buildPairSummaryRows groups by pair; drillToPartner shim throws in dev to enforce.*
- [x] **PCFG-05**: Each `(partner, product)` pair has an editable segment list (named sub-cohorts like "English", "Spanish", "Bank A", "Bank B"), persisted in localStorage. Partner Setup UI reachable from the sidebar partner context menu edits only segments — product type is not editable because it's data-derived. — *Plan 39-02: src/lib/partner-config/* domain (types, schema, storage, evaluator, viable-columns) + usePartnerConfig hook + PartnerConfigProvider context + PartnerSetupSheet slide-over + ContextMenu wiring. localStorage key `bounce-dv-partner-config`. Validation: empty/duplicate/'Other' hard-block, overlap warns with force-confirm. Today's batch-summary has no viable segmenting columns — Setup shows empty-state Alert; scaffolding ready for ETL adds.*
- [x] **PCFG-06**: `PartnerListFilters` is extended with `PRODUCT_TYPE` (auto-derived from `ACCOUNT_TYPE`) and `SEGMENT` (from config) attribute filters. Pre-populated Partner Lists auto-maintain one list per distinct product-type value, derived from data. — *Plan 39-03: PartnerListFilters gains optional PRODUCT_TYPE + SEGMENT keys; .strict() schema preserved with additive-optional evolution; PartnerList.source variant 'derived'; computeDerivedLists generates one list per distinct ACCOUNT_TYPE with stable IDs (DERIVED_LIST_ID_PREFIX); usePartnerLists merges derived lists without persisting; sidebar visual distinction = Sparkles + "Auto" pill; rename disabled with Tooltip; sonner "reappears on refresh" toast on derived-delete; filter-evaluator handles PRODUCT_TYPE alias + SEGMENT via optional resolver; create-list-dialog wires segmentResolver from usePartnerConfigContext.*
- [x] **PCFG-07**: Charts and KPIs accept an optional segment dimension — when a `(partner, product)` pair has segments configured, the user can choose to split curves/KPIs by segment instead of rolling up. Pairs without segments configured render as the pair-rolled-up view (segment-level zero-regression — but note: pair-level splitting is NOT opt-in, it's enforced). — *Plan 39-04: src/lib/partner-config/segment-split.ts centralizes splitRowsBySegment / kpiAggregatesPerSegment / reshapeCurvesPerSegment / averageCurvesPerSegment / tagRowsWithSegment with apples-and-oranges invariant smoke (10 assertion blocks). Independent per-view toggles on Collection Curve chart (Switch in actions slot, synthetic dollar-weighted-average curve per segment) + KPI block (grouped layout, one cascade row per segment, Other muted via bg-surface-inset). Chart Builder Segment series option gated on hasAnySegments. GenericChart useSegmentTaggedRows handles single-pair + multi-pair scope (per-pair segment resolution with displayName fallback for unconfigured pairs). PartnerStats.rawRows additive field exposes pair-filtered row set (Pitfall 7 — single SoT). Stale-column resolver allowlists '__'-prefixed sentinels.*
