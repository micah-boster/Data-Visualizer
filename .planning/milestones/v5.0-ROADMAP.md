# Milestone v5.0: External Intelligence

**Status:** PLANNED (not started — follows v4.5)
**Phases:** 45-49
**Theme:** "See the full picture"

## Overview

The tool stops being an internal-only dashboard and becomes a competitive intelligence platform. Three external data sources come in — partner scorecards, contractual targets, and industry context — and the comparison views that triangulate all three transform every partner conversation.

After v4.5 hardens the foundation, v5.0 makes the tool *irreplaceable* — no off-the-shelf tool gives you internal performance + partner-reported metrics + contractual targets in a single view.

**Success criteria:** MBR prep starts in this tool. Partner conversations are grounded in triangulated data — your numbers, their numbers, the targets — not just one perspective.

## Phases

### Phase 45: Scorecard Ingestion Pipeline

**Goal**: Upload a partner scorecard in any format (PDF, Excel, CSV, email paste), extract structured metrics via Claude, confirm/correct the mapping, and store alongside internal data
**Depends on**: v4.0 complete (design system, component patterns), v4.1 complete (Partner Config module — scorecards associate with partner + segment), v4.5 complete (hardening foundation)
**Effort**: Large (multi-format extraction, per-partner schema learning, normalization layer)
**Requirements**: SC-01 through SC-09

**Success Criteria** (what must be TRUE):
  1. User can upload a PDF scorecard and Claude extracts key metrics with field labels
  2. User can upload an Excel/CSV scorecard and the app parses structured columns automatically
  3. User can paste email body text and Claude extracts embedded metrics
  4. First scorecard for a partner requires manual field mapping (their metric names → our vocabulary)
  5. Subsequent scorecards from the same partner auto-map using the learned schema
  6. Every extracted value shows a confidence indicator; low-confidence values are flagged for human review
  7. User confirms/corrects every extraction before it's stored (human-in-the-loop, v1)
  8. Scorecard data is stored with partner ID, date, source format, and extraction metadata
  9. Scorecard library shows all historical scorecards per partner, searchable by date

**Automation graduation path** (not in v5.0 scope, noted for future):
- Phase 1 (v5.0): Human reviews every scorecard extraction
- Phase 2 (v5.5+): Auto-ingest when confidence is high, queue for review when it's not
- Phase 3 (v6+): Fully automatic with exception-only review

**After Phase 45, kick off simultaneously:**

### Phase 46: Contractual Target Management

**Goal**: Enter, extract, and manage per-partner per-metric targets from contracts and internal goals. Targets become a persistent reference layer across all views.
**Depends on**: Phase 45 (shares the external data model and normalization vocabulary)
**Effort**: Medium (target entry UI, contract PDF extraction, date-range versioning)
**Requirements**: TGT-01 through TGT-06

**Success Criteria** (what must be TRUE):
  1. User can manually enter targets per partner per metric with effective date ranges
  2. User can upload a contract PDF and Claude extracts target clauses for confirmation
  3. Targets are typed: hard SLA (contractual), soft target (internal goal), or industry benchmark
  4. Targets are versioned — when a contract renews with new targets, the old targets are archived, not overwritten
  5. Target data is accessible to charts, tables, and Claude NLQ context
  6. Editing a target shows the change history (who changed what, when)

**Plans**: _To be scoped via `/gsd:plan-phase` when this phase is ready to start. Plan count and breakdown depend on codebase state and outcomes from prior phases._

**After Phase 46:** Feeds into Phase 47 (triangulation views become fully powered with all three data sources).

### Phase 47: Triangulation Views

**Goal**: Side-by-side comparison of internal metrics vs. partner-reported scorecard vs. contractual targets — the killer feature that changes every partner conversation
**Depends on**: Phase 45 (scorecard data). Enriched by Phase 46 (targets) but can ship with scorecards only.
**Effort**: Medium-Large (new view mode, divergence logic, traffic-light system, trend overlays)
**Requirements**: TRI-01 through TRI-08

**Success Criteria** (what must be TRUE):
  1. Partner detail view shows a triangulation panel: internal metric, scorecard metric, and target side-by-side for each comparable metric
  2. Divergence highlighting: metrics where internal and scorecard values disagree by more than a configurable threshold are visually flagged
  3. Target traffic-light per metric: green (on/above target), amber (within X% of target), red (below target)
  4. Overall partner health indicator: aggregate traffic-light across all target metrics
  5. Trend view: show internal vs. scorecard vs. target over time (are gaps widening or closing?)
  6. Missing data handling: graceful display when scorecard or target data is unavailable for a metric
  7. Collection curve chart can overlay target projection line alongside internal curves
  8. Claude NLQ can answer questions that reference all three data sources ("How does Partner X's reported recovery rate compare to our measurement and the contractual target?")

**Plans**: _To be scoped via `/gsd:plan-phase` when this phase is ready to start. Plan count and breakdown depend on codebase state and outcomes from prior phases._

**After Phase 47, kick off:**
- **Phase 48** (Scorecard Reconciliation) — deeper analysis of the scorecard relationship over time

### Phase 48: Scorecard Reconciliation & History

**Goal**: Track the relationship between internal and partner-reported data over time — who's consistently accurate, where do gaps persist, and how does the relationship evolve?
**Depends on**: Phase 47 (triangulation views — reconciliation is the temporal extension)
**Effort**: Medium (reconciliation dashboard, drift detection, reliability scoring)
**Requirements**: REC-01 through REC-05

**Success Criteria** (what must be TRUE):
  1. Reconciliation dashboard per partner: timeline of scorecard submissions aligned with internal data snapshots
  2. Drift detection: flag when the gap between scorecard and internal metrics changes significantly (e.g., divergence grew from 2% to 8% over 3 months)
  3. Partner reliability indicator: which partners' scorecards consistently match internal data? Summarized as a reliability tier per partner.
  4. Metric-level reconciliation: some metrics may agree while others diverge — show per-metric alignment history
  5. Reconciliation summary exportable for partner conversations (data package with charts)

**Plans**: _To be scoped via `/gsd:plan-phase` when this phase is ready to start. Plan count and breakdown depend on codebase state and outcomes from prior phases._

**After Phase 48:** No new phases unlocked. Feeds into v5.0 milestone completion.

### Phase 49: Dynamic Curve Re-Projection

**Goal**: Collection curves project forward using actual collection data, contractual targets, and partner-reported trajectories — not just historical averages. Extends v4.1 Phase 40's static historical-average projection.
**Depends on**: Phase 46 (targets provide projection anchors), Phase 47 (triangulation infrastructure), v4.1 Phase 40 (Projected Curves v1 foundation)
**Effort**: Medium (forecasting model + chart integration + confidence bands)
**Requirements**: PROJ-01 through PROJ-04

**Success Criteria** (what must be TRUE):
  1. Projected curves re-calculate as actual collection data arrives (not static from day 1 — extends v4.1 Phase 40's static projection)
  2. Three projection overlays available: "at current pace", "to hit target", "partner-reported trajectory"
  3. Confidence bands show projection uncertainty (wider with less data, narrower with more)
  4. Projections respect the anomaly detection system — flagged batches show adjusted projections

**Plans**: _To be scoped via `/gsd:plan-phase` when this phase is ready to start. Plan count and breakdown depend on codebase state and outcomes from prior phases._

**After Phase 49:** No new phases unlocked. Feeds into v5.0 milestone completion.

## Key Decisions

| Decision | Rationale |
|----------|-----------|
| Human-in-the-loop on every scorecard (v5.0) | 5-15 partners, high stakes data. Graduate to auto-ingest in v5.5+ once schemas are proven. |
| Per-partner schema learning | Different partners use different metric names and formats. Learn once, auto-map thereafter. |
| Triangulation as the centerpiece view | This is the single most differentiated feature. No off-the-shelf tool does internal + external + target comparison. |
| Dynamic curves require targets (Phase 46) | Projecting "to hit target" is meaningless without targets. Curve re-projection deferred until target data exists. |
| Phase 49 extends v4.1 Phase 40 | v4.1 ships a static historical-average projection to benchmark against now; Phase 49 re-projects with actuals, targets, and confidence bands on top of that foundation. |
| Scorecard data stored with extraction metadata | Auditability matters — always know where a number came from, what format, what confidence. |
