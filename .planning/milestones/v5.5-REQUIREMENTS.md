# Requirements: v5.5 Real-Use Hardening

**Defined:** 2026-04-21
**Core Value:** Post-v5.0 hardening that improves when deferred — behavioral QA written from observed MBR-prep workflows (not hypothetical scripts) and tech-debt sweep informed by v5.0's actual code additions. Complements v4.5's pre-v5.0 structural audits (correctness + security architecture).

## Theme

v5.5 holds the audits whose value grows with observation. Splitting the hardening work across v4.5 (load-bearing for v5.0) and v5.5 (informed by v5.0) lets each piece land at its most effective moment. v4.5 prevents v5.0 from shipping on shaky ground; v5.5 consolidates what v5.0 actually produced.

## Behavioral QA from Usage (Phase 50)

**Principle:** A QA script written before v5.0 ships audits hypothetical flows. A QA script written after v5.0 is in daily MBR-prep use audits what actually breaks. The latter is worth more per hour invested. v5.5 observes first, authors second. **Critical**: the in-situ instrumentation (IUR-01..03 below) must be built into v5.0 *as it ships*, not bolted on in v5.5 — without telemetry and a feedback channel, "observation" becomes "ask people sometimes how it's going" which is no data at all.

- [ ] **QA-01**: `docs/QA-SCRIPT.md` exists as a living checklist the team runs before each milestone ships. Every entry cites the real usage or failure mode that motivated it — no hypothetical scenarios.
- [ ] **QA-02**: Regression test battery covers v5.0 features explicitly: scorecard ingestion paths (PDF, Excel, CSV, email), triangulation views under all data-completeness combinations (all three sources, missing scorecard, missing target, missing both), dynamic curves with various actuals-windows and anomaly overlays.
- [ ] **QA-03**: Keyboard + mouse regression script covers end-to-end drill flows (root → partner → `(partner, product)` → batch → account) under both single-product and multi-product partners, with filters applied and cleared. Matches the Phase 39 scope change.
- [ ] **QA-04**: Dark/light mode parity check covers all v5.0 surfaces — triangulation panels, scorecard library, target management UI, reconciliation dashboard.
- [ ] **QA-05**: Cross-browser smoke on core flows — Chrome, Safari, Firefox at minimum. Browser-specific bugs are fixed or documented with workarounds.
- [ ] **QA-06**: Screen-size parity check — laptop (≤900px height per v4.1 CHT-04), desktop, and tablet if relevant to actual use.

## In-situ User Research Infrastructure (Phase 50, co-built with v5.0)

**Principle:** Real-use observation requires real-use instrumentation. These deliverables are built into v5.0 as it ships, not retrofitted in v5.5. They produce the data Phase 50's QA script consumes.

- [ ] **IUR-01**: Lightweight event telemetry. `/api/events` sink records `(user, timestamp, event, context)` tuples. Instrumented moments: filter applied, drill performed, view saved, view loaded with stale columns, `⌘K` opened, anomaly clicked, scorecard imported (v5.0), triangulation viewed (v5.0), confusion-button submitted. Persistence to a Snowflake events table or flat-file in S3 — internal-tool grade, no PII concern.
- [ ] **IUR-02**: "This is confusing" button in the help drawer. One click captures current URL, filters, drill state, viewport, theme, opens a textarea. Submit posts to a Slack channel or Notion DB. Cost to user: 10s. Cost to team to read: 30s/entry.
- [ ] **IUR-03**: Weekly 15-minute review ritual, calendared. Skim the week's telemetry, read every confusion submission, pick 1-2 for the next phase's plan intake. Without the calendar block this never happens.

## Tech Debt Sweep (Phase 51)

**Principle:** Debt accumulated through v4.1/v4.5/v5.0 is triaged and closed in one pass. Sweeping earlier misses v5.0's additions; sweeping later compounds with v6.0's. The specific target is code duplication — v5.0 will likely re-implement logic that already exists in `src/lib/computation/*` for scorecards, triangulation, and dynamic curves. This phase consolidates.

- [ ] **DEBT-01**: Every `docs/KNOWN-ISSUES.md` item carried forward from v3.1/v4.0/v4.1/v4.5/v5.0 is either closed or explicitly triaged with a rationale for deferral. Empty backlog (or an all-triaged one) is the goal.
- [ ] **DEBT-02**: Code duplication between v5.0's new computation (scorecard normalization, triangulation math, dynamic-curve projection) and the existing `src/lib/computation/*` layer is consolidated. If scorecards re-implement anything internal metrics already do, they share it.
- [ ] **DEBT-03**: TanStack v9 migration (KI-13, KI-14) is either executed or explicitly scheduled for a later milestone with the upgrade path documented.
- [ ] **DEBT-04**: Major dependency upgrades flagged during v4.5 SEC-06 are executed or triaged with risk notes. No silent drift into unmaintained versions.
- [ ] **DEBT-05**: Dead code, deprecated patterns, and stale abstractions discovered during v4.5 audits and v5.0 development are retired.
- [ ] **DEBT-06**: Hot-path code identified as slow during real v5.0 use (large scorecard uploads, triangulation with many metrics, dynamic curve recalculation at scale) is profiled and either optimized or triaged with a performance budget.
- [ ] **DEBT-07**: Decompose `data-display.tsx` (currently ~1458 lines, ~38 hooks) into section components — `DataDisplayShell` (layout only, <150 lines) + `ChartSection` / `TableSection` / `KpiSection` / `DrillSection` (each <300 lines, each independently testable). Pull filter/drill/chart orchestration into a typed `useDashboardMachine` (XState or tagged reducer). Component-size CI gate (`check:component-size`): max <300 lines.
- [ ] **DEBT-08**: State consolidation — collapse the 5 React Contexts (`sidebar-data`, `partner-lists`, `active-partner-list`, `partner-norms`, `anomaly-provider`, `data-freshness`, `cross-partner-provider`) plus ~38 local `useState` hooks plus URL params plus localStorage into a single typed Zustand store with named slices. Server cache stays in React Query. Optional dev-only `<StateInspector />` overlay (toggle with `⌘⇧D`) renders the store tree.
- [ ] **DEBT-09**: Test pyramid inversion — add Vitest, port the 7 `*.smoke.ts` scripts into real test files, target ≥95% line / ≥90% branch coverage on `src/lib/computation/**`, `src/lib/columns/**`, `src/lib/data/**` (post-BND-01), `src/lib/views/**`. Property-based tests via fast-check on `coerceAgeMonths`, `reshape-curves`, `compute-trending`, view migrations (round-trip property). Component interaction tests for `DataTable` sorting, filter popover, drill, saved-view load/save. Contract tests for `/api/data` schema. Error-path tests injecting 500s, empties, malformed rows.
- [ ] **DEBT-10**: Performance budget enforced in CI — `@next/bundle-analyzer` per-route budget (e.g., initial JS ≤180 KB gz; calibrated against v5.0's actual bundle). Lighthouse CI on deployed preview: perf ≥90, a11y ≥98, best-practices ≥95. `why-did-you-render` in dev. Web Vitals reported from `app/layout.tsx` to `/api/vitals` sink. PRs display budget delta; >5% regression requires explicit override.
