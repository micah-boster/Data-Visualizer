# Milestone v4.5: Correctness & Security Foundation

**Status:** PLANNED (not started — follows v4.1, precedes v5.0)
**Phases:** 41-42 (43-44 freed as insert-phase slack)
**Theme:** "Structural guardrails before external data arrives"

## Overview

v4.5 is the tight, structural hardening pass that must precede v5.0. Its scope is deliberately narrow: data correctness on the metrics triangulation will compare, and security review on both the existing surface and the forward threat model for v5.0's ingestion pipeline. Behavioral QA and tech-debt sweep are **not** in v4.5 — those live in v5.5 where post-usage observation can inform them.

The ordering is non-negotiable: if v5.0 triangulation ships on top of silent correctness bugs, "partner disagrees with our data" becomes an unrecoverable trust problem (in both the tool and the partner relationship). If v5.0 ingestion ships before its threat model is known, architectural security issues bake in. Both risks degrade when deferred.

**Success criteria:** Every number visible in the app matches what a direct Snowflake query would return. The `(partner, product)` scope change from Phase 39 produces consistent numbers at every drill level. v5.0's file-upload + Claude-extraction surface has a documented threat model before Phase 45 plans it.

## Phases

### Phase 41: Data Correctness Audit

**Goal**: Systematic audit of every metric that v5.0 triangulation will eventually compare — collection rates, commitment rate (once data lands), KPI math, chart series math, and scope rollups at every drill level. Close all correctness bugs, fix the seed bug, and lock in a regression contract.
**Depends on**: v4.1 complete (Phase 39's `(partner, product)` scope change must be stable before auditing rollups; Phase 38's KPI cascade must be settled before auditing KPI math)
**Effort**: Medium (seed bug is scoped; the audit is breadth-first across existing computations)
**Requirements**: DCR-01 through DCR-06

**Success Criteria** (what must be TRUE):
  1. Summary row seed bug is fixed (summary row aggregation + formatter binding — per-column aggregation strategy: `sum` / `avgWeighted` / `none` / `range`; footer reuses each column's own formatter)
  2. Every metric displayed in the app (KPI cards, chart series, table cells, summary rows, partner rollups) is verified against a direct Snowflake query at three scope levels: root (all partners), `(partner, product)` pair, and batch
  3. Scope rollups are consistent across drill levels — selecting a partner-product pair produces the same KPI values as filtering the root table to that pair and reading its summary row
  4. Every correctness bug found gets a regression fixture or smoke test so it cannot silently return
  5. Aggregation contract is documented in `docs/` (when to `sum`, when to weight by volume, when to suppress aggregation) so future column additions don't re-break the rule
  6. Audit report enumerates every metric touched and its verification status — becomes the correctness baseline v5.0 triangulation is built on

**Plans**: _To be scoped via `/gsd:plan-phase` when this phase is ready to start. Likely 2-3 plans: seed-bug fix + per-column aggregation strategy, breadth-first metric audit, regression test battery + docs._

### Phase 42: Ingestion-Surface Security Review

**Goal**: Audit the existing security surface (credentials, API routes, client-side exposure, SQL injection) AND produce a forward threat model for v5.0's file-upload + Claude-extraction + stored-scorecard surface so Phase 45 architects ingestion with the threat model already known.
**Depends on**: None within v4.5 (can run in parallel with Phase 41)
**Effort**: Medium (audit + threat model write-up; findings drive Phase 45 architecture)
**Requirements**: SEC-01 through SEC-06

**Success Criteria** (what must be TRUE):
  1. Credential handling audit complete — Snowflake + Anthropic API keys verified server-only, no client-side exposure via sourcemaps, bundles, or network traces
  2. Client-side data exposure audit complete — documented what's reachable via devtools, localStorage, sourcemaps, and the network tab; any unintended exposure (e.g., PII in network traces) is fixed
  3. SQL injection surface review complete — `/api/data`, `/api/accounts`, Metabase import parser, and any NLQ context construction verified against the column allow-list pattern
  4. Forward threat model for v5.0 Phase 45 documented — covers: uploaded file validation (MIME, size, extension), PDF/Excel parsing sandbox, Claude extraction prompt-injection risk, stored scorecard PII handling, rate limits, auth model for uploads
  5. Auth and access control state documented — current model (if any) explicit, v5.0 requirements explicit, gap analysis feeds Phase 45 planning
  6. Dependency security baseline established — `npm audit` clean or triaged, Dependabot or equivalent configured, major-version upgrade paths noted for v5.5 tech-debt sweep

**Plans**: _To be scoped via `/gsd:plan-phase` when this phase is ready to start. Likely 2-3 plans: existing-surface audit, forward threat model for ingestion, dependency + auth baseline._

## Key Decisions

| Decision | Rationale |
|----------|-----------|
| v4.5 scoped to correctness + security only | Both risks degrade when deferred — silent bugs compound, architectural security issues bake in. Behavioral QA and tech-debt sweep improve when deferred, so they move to v5.5. |
| 43-44 reserved as insert-phase slack | Useful for `/gsd:insert-phase` urgent work discovered during v5.0. Keeps v5.0 phase numbers (45-49) stable so inbound references don't churn. |
| Phase 41 audit is breadth-first, not deep-dive | Goal is to find every correctness bug affecting triangulation, not to rewrite the computation layer. Depth-first rewrites are tech-debt work and belong in v5.5. |
| Phase 42 produces a forward threat model | v5.0 Phase 45 architecture is significantly cheaper to get right than to retrofit — ingestion surface (file upload, Claude extraction) has well-known failure modes that should shape design, not patch it later. |
| Behavioral QA moved to v5.5 | A QA script written before v5.0 ships audits hypothetical flows. Real MBR-prep usage reveals what actually matters. Build the script from observed use, not imagined use. |
