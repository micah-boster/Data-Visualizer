## Delivered (one-line summary)

Every visible number now has a Snowflake-equivalence row in `docs/METRIC-AUDIT.md`; the compute layer reads `BatchRow[]` (typed, branded age, NULL-aware rate fields) instead of bag-of-strings; persistence is versioned with explicit migration / verified-write / cross-tab sync; Snowflake calls retry with circuit-breaker + request-id correlation + Server-Timing; charts compose from a single `<ChartFrame>` primitive with polarity context; v5.0's ingestion surface has a load-bearing forward threat model (SEC-04, 13 LOCK mitigations) that Phase 45 will consume; and 15 domain terms live in a single `TERMS` registry with `<Term>` popovers wired into KPIs / breadcrumbs / sidebar / table headers, including REVENUE_MODEL as the third unit-of-analysis dimension.

## Overview

v4.5 is the tight, structural pass that must precede v5.0. Its scope is the work whose value *degrades* if deferred past v5.0 (in contrast to v5.5, which holds work whose value *grows* with post-v5.0 observation).

Originally scoped to 2 phases (Data Correctness Audit + Security Review). A multi-lens audit in late April 2026 (data, structural/systems, design-thinking, code-quality reviews) surfaced two additional categories of work that share the same degradation profile and must land before v5.0:

- **Boundary Hardening (new Phase 43)** — v5.0 adds 3 new data shapes (scorecards, targets, triangulation rows). Without typed parsing boundaries, persistence schema versioning, and a unified chart primitive, v5.0 inherits and amplifies current representation debt × 3.
- **Vocabulary Lock (new Phase 44)** — v5.0 adds 5 new domain terms (scorecard, target, triangulation, reconciliation, divergence) on top of 12 existing unowned concepts. A glossary and term registry must precede new term introduction or the conceptual sprawl compounds.

Wave 0 (three quick correctness fixes from the data review) shipped 2026-04-26 in advance of Phase 41 starting: MIN_GROUPS gate on anomaly detection, matrix default to bar mode, KPI suppression on insufficient 3mo denominator.

**Success criteria:** Every number visible in the app matches a direct Snowflake query. The `(partner, product)` scope change from Phase 39 produces consistent numbers at every drill level. v5.0's file-upload + Claude-extraction surface has a documented threat model. The compute layer operates on typed canonical rows, not bag-of-strings. localStorage state survives schema changes with explicit, repairable feedback. The first 17 domain terms (12 existing + 5 v5.0) are defined once and surfaced in-product.

**Success Criteria** (what must be TRUE):
  1. Summary row seed bug is fixed (summary row aggregation + formatter binding — per-column aggregation strategy: `sum` / `avgWeighted` / `none` / `range`; footer reuses each column's own formatter)

### Phase 41.2: v4.1 Carryover Stabilization ✅ 2026-04-27

**Status:** Complete. Both bugs root-caused and fixed without a `/gsd:debug` session — static analysis localized both to the same Phase 39 PCFG-04 oversight (multi-product-partner keying). No PLAN.md ceremony.

**Goal:** Fix two unrelated bugs surfaced during Phase 40.1 browser UAT that were deferred to keep the v4.1 milestone close clean.

**Depends on**: Independent — can run before or after Phase 41. Practically should land before Phase 41 since the duplicate-key bug may be in the in-flight Phase 41-pre Wave 0 code (anomaly compute changes).

**Effort**: Small. Bug (a) likely needs `/gsd:debug` first — root cause not localizable via static analysis. Bug (b) is a static fix once scope is confirmed.

**Bugs in scope:**
1. **React duplicate-key `'Happy Money'`** — fires from a re-render path under `DataDisplay` (trace ends at `flushPassiveEffects → DataDisplay`). Static grep over `key={` patterns showed all sites are pairKey-keyed; the collision must be in a less obvious render path. Suspects: `anomaly-provider.tsx` enrichment merge, or one of the in-flight Phase 41-pre Wave 0 changes (`compute-anomalies.ts`, `kpi-summary-cards.tsx`, `compute-kpis.ts`, `comparison-matrix.tsx`).
   - **Actual cause:** `RootSparkline` (`src/components/charts/root-sparkline.tsx`) keyed pivoted Recharts data + `lineKeys` on bare `partnerName`. Multi-product partners (Happy Money, Zable) emit two `CrossPartnerEntry` rows that collapse to one `point["Happy Money"]` and duplicate `lineKeys` entries — Recharts then renders two `<Line key="Happy Money">` siblings.
   - **Fix:** switch to `pairKey({ partner, product })` for both data keys and `lineKeys`. Commit `5083c88 fix(41.2): use pairKey in RootSparkline data keys`.
2. **4 always-empty trailing footer cells at All Partners root drill** — `buildRootColumnDefs()` returns 9 data cols + 1 anomaly = 10, but observed footer renders ~12 cells with 4 trailing blanks. Possible regression on Plan 40.1-03 truth #5 (modeled+delta cols leaking past the `columnDefsOverride` boundary at root) OR pre-existing column-picker state showing stale columns.
   - **Actual cause:** the 4 trailing cells are the percentile-rank columns (`Pen. Rank`, `6mo Rank`, `12mo Rank`, `Collected Rank`) added at root via `extraColumns` in `data-table.tsx`. Two compounding issues: (a) `percentile-columns.tsx` looked up `cpData.partners.get(partnerName)` but `cpData.partners` is keyed by `pairKey` (PCFG-04) — every row returned `undefined`, rendering "—" in the body; (b) the column had no `meta.type`, so `table-footer.tsx` skipped them as text columns and rendered blank instead of the existing null-aggregate em-dash.
   - **Scope amendment:** "exactly 10 columns" criterion replaced with "rank columns work and footer renders the existing null-aggregate em-dash" — keeping the rank feature was cheaper than ripping it out and matched the codebase's existing footer convention. User-confirmed during planning.
   - **Fix:** switch lookup to `pairKey`, declare `meta: { type: 'count', aggregation: 'none' }`. Folded into commit `9f5e0ec fix(41-01): wire dollar-weighted penetration + audit column strategies`.

**Success Criteria** (what must be TRUE):
1. ✅ Console at root drill ("All Partners" view) is clean — no React duplicate-key warnings, no other reconciler errors
2. ✅ Root drill table renders all visible columns with no broken-looking footer cells; rank columns render `Pxx (n/N)` badges in body and "—" in footer (amended from "exactly 10 columns" — see Bug 2 scope amendment above)
3. Pending UAT verification — no regression on Phase 40.1's 8 observable truths

**Plans**: None — fixed inline via two atomic commits.

### Phase 43: Boundary Hardening ✅ SHIPPED 2026-05-01

**Plans (4/4):** 43-01 canonical `BatchRow`/`AccountRow` + branded `BatchAgeMonths` + long-format `CurvePoint` + 8 compute files migrated, three duplicate `coerceAgeMonths` collapsed (BND-01/02); 43-02a `createVersionedStore<T>` with envelope/migrations/verified-write/cross-tab subscribe across 5 storage modules (BND-03); 43-02b `executeWithReliability` with retry + circuit breaker + `Server-Timing` + `X-Request-Id` + `<DegradedBanner>` + `(stale)` badge (BND-04); 43-03 `<ChartFrame>` primitive with polarity context + tuned `unstable_cache` + `<RefreshButton>` + ADR 009 (BND-05/06).

**Goal**: Establish typed parsing boundaries, versioned persistence, Snowflake reliability primitives, a unified `<ChartFrame>` primitive, and tuned caching — so v5.0 inherits a typed substrate, lossless persistence, graceful failure modes, and a single chart system rather than amplifying current debt with three new data shapes.
**Depends on**: Phase 41 partially (DCR-08 NULL semantics is co-implemented with BND-01 canonical row representation — same parser does both). Independent of Phase 42.
**Effort**: Medium-Large (4-5 days; biggest of the v4.5 phases because it touches the most seams)
**Requirements**: BND-01 through BND-06

**Success Criteria** (what must be TRUE):
  1. `BatchRow` and `AccountRow` typed interfaces with resolved types — numbers as `number | null` for rate-shaped fields, branded `BatchAgeMonths`, long-format curves baked in. Parser at `src/lib/data/parse-batch-row.ts`. Static cache loads through the same parser.
  2. Compute layer signature change — every function in `src/lib/computation/` takes the typed row, not `Record<string, unknown>[]`. The three duplicate `coerceAgeMonths` functions collapse to one (the parser).
  3. Persistence layer with `_meta: { schemaVersion, savedAt }` envelope on every blob, migration chain, verified writes (read-back-and-compare), cross-tab `storage` event listener. Replaces direct localStorage in views/columns/lists/presets.
  4. Snowflake reliability — exponential-backoff retry on transient errors, queue-wait separated from query-execution time, request-id correlation (`X-Request-Id` header), sanitized client errors, lightweight circuit-breaker (5 fails / 30s → 60s degraded mode using stale React Query cache).
  5. `<ChartFrame>` primitive — single component owning title, legend slot, tooltip slot, empty/loading/error states, stale-column warning, density-aware padding, polarity prop. All four current charts compose from it. v5.0's triangulation visualizations inherit the frame.
  6. Caching tuned, not disabled — `revalidate: 3600` + tag-invalidation on the data API; React Query `staleTime: 5min`, no refetch on focus; manual Refresh affordance; ADR documenting all three layers.
  7. Snowflake credit consumption drops by ≥10× in the week after deployment (measurable from warehouse usage view) — caching pays for itself.

**Plans**: _To be scoped via `/gsd:plan-phase`. Likely 3 plans: (a) canonical row representation + compute-layer signature change + NULL semantics co-implementation (BND-01/02, with DCR-08), (b) versioned persistence + Snowflake reliability primitives (BND-03/04), (c) ChartFrame primitive + caching tuning (BND-05/06)._

## Key Decisions

| Decision | Rationale |
|----------|-----------|
| v4.5 expanded from 2 to 4 phases (2026-04-26) | Multi-lens audit surfaced two more categories with the same "degrades when deferred" profile: boundary hardening (v5.0 adds 3 new data shapes; without typed parsers and unified chart primitive, debt compounds × 3) and vocabulary lock (v5.0 adds 5 new terms; without a glossary first, conceptual sprawl compounds). Both are pre-v5.0 work by the same logic that put correctness and security pre-v5.0. |
| Phases 43-44 use the reserved slack slots | The 43-44 slots were originally reserved for `/gsd:insert-phase` urgent work during v5.0. Reallocating them to defined v4.5 phases is fine; v5.0 phase numbers (45-49) remain stable. |
| Phase 41 audit is breadth-first, not deep-dive | Goal is to find every correctness bug affecting triangulation, not to rewrite the computation layer. Depth-first rewrites are tech-debt work and belong in v5.5 (DEBT-07..10). |
| Phase 41 statistical fixes are targeted, not architectural | DCR-07 through DCR-11 are surgical fixes to known issues identified in the data review. The architectural rewrite (e.g., Bayesian anomaly detection, t-distribution for small samples) is out of scope. |
| Phase 42 produces a forward threat model | v5.0 Phase 45 architecture is significantly cheaper to get right than to retrofit — ingestion surface (file upload, Claude extraction) has well-known failure modes that should shape design, not patch it later. |
| Phase 42 split into 42a + 42b on 2026-04-30 | OAuth still not configured. ~70% of Phase 42 (SEC-01/03/04/06) is OAuth-independent and includes the load-bearing forward threat model. Splitting lets v4.5 close cleanly with 42a; SEC-02/05 (the parts that genuinely need a deployed authenticated surface) become 42b and slot in whenever OAuth lands. SEC-04 specifically must NOT slip past Phase 45 start regardless of OAuth timing — the forward threat model is what shapes ingestion architecture. |
| Phase 43 ChartFrame is a primitive, not a full chart-system rebuild | Lift the common shell (legend / tooltip / empty state / polarity) into one component. Each existing chart re-skins to compose from it. The full chart-grammar redesign is a v6.0 design pass. |
| Phase 44 ships before v5.0 begins | v5.0 introduces 5 new domain terms. Locking vocabulary after v5.0 means retrofitting `<Term>` wrappers everywhere. Locking before means v5.0 plans extend `TERMS` as part of their deliverable. |
| Behavioral QA moved to v5.5 | A QA script written before v5.0 ships audits hypothetical flows. Real MBR-prep usage reveals what actually matters. Build the script from observed use, not imagined use. |
| Component decomposition / state consolidation moved to v5.5 | The right seams to extract are the ones v5.0 reveals. Premature decomposition creates abstractions v5.0 then needs to rework. |
