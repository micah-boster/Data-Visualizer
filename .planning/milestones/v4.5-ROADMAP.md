# Milestone v4.5: Correctness & Foundation

**Status:** PLANNED — Wave 0 quick fixes shipped 2026-04-26; Phases 41-44 not yet started
**Phases:** 41-44 (originally 41-42; 43-44 activated 2026-04-26 from reserved slack)
**Theme:** "Structural guardrails before external data arrives"

## Overview

v4.5 is the tight, structural pass that must precede v5.0. Its scope is the work whose value *degrades* if deferred past v5.0 (in contrast to v5.5, which holds work whose value *grows* with post-v5.0 observation).

Originally scoped to 2 phases (Data Correctness Audit + Security Review). A multi-lens audit in late April 2026 (data, structural/systems, design-thinking, code-quality reviews) surfaced two additional categories of work that share the same degradation profile and must land before v5.0:

- **Boundary Hardening (new Phase 43)** — v5.0 adds 3 new data shapes (scorecards, targets, triangulation rows). Without typed parsing boundaries, persistence schema versioning, and a unified chart primitive, v5.0 inherits and amplifies current representation debt × 3.
- **Vocabulary Lock (new Phase 44)** — v5.0 adds 5 new domain terms (scorecard, target, triangulation, reconciliation, divergence) on top of 12 existing unowned concepts. A glossary and term registry must precede new term introduction or the conceptual sprawl compounds.

Wave 0 (three quick correctness fixes from the data review) shipped 2026-04-26 in advance of Phase 41 starting: MIN_GROUPS gate on anomaly detection, matrix default to bar mode, KPI suppression on insufficient 3mo denominator.

**Success criteria:** Every number visible in the app matches a direct Snowflake query. The `(partner, product)` scope change from Phase 39 produces consistent numbers at every drill level. v5.0's file-upload + Claude-extraction surface has a documented threat model. The compute layer operates on typed canonical rows, not bag-of-strings. localStorage state survives schema changes with explicit, repairable feedback. The first 17 domain terms (12 existing + 5 v5.0) are defined once and surfaced in-product.

## Phases

### Phase 41: Data Correctness Audit

**Goal**: Systematic audit of every metric that v5.0 triangulation will eventually compare — collection rates, commitment rate (once data lands), KPI math, chart series math, scope rollups at every drill level — PLUS the targeted statistical fixes surfaced by the late-April 2026 data review (young-batch censoring, NULL-vs-zero semantics, polarity consistency, runtime invariant for apples-and-oranges, threshold ADRs). Close all correctness bugs, fix the seed bug, and lock in a regression contract.
**Depends on**: v4.1 complete (Phase 39's `(partner, product)` scope change must be stable before auditing rollups; Phase 38's KPI cascade must be settled before auditing KPI math). Wave 0 fixes (already shipped 2026-04-26) are referenced by DCR-07/08/11.
**Effort**: Medium-Large (originally Medium for breadth-first audit; expanded scope adds the data-review fixes which are targeted rather than rewrites)
**Requirements**: DCR-01 through DCR-11 (originally DCR-01..06; DCR-07..11 added 2026-04-26)

**Success Criteria** (what must be TRUE):
  1. Summary row seed bug is fixed (summary row aggregation + formatter binding — per-column aggregation strategy: `sum` / `avgWeighted` / `none` / `range`; footer reuses each column's own formatter)
  2. Every metric displayed in the app (KPI cards, chart series, table cells, summary rows, partner rollups) is verified against a direct Snowflake query at three scope levels: root (all partners), `(partner, product)` pair, and batch
  3. Scope rollups are consistent across drill levels — selecting a partner-product pair produces the same KPI values as filtering the root table to that pair and reading its summary row
  4. Every correctness bug found gets a regression fixture or smoke test so it cannot silently return
  5. Aggregation contract is documented in `docs/` (when to `sum`, when to weight by volume, when to suppress aggregation) so future column additions don't re-break the rule
  6. Audit report enumerates every metric touched and its verification status — becomes the correctness baseline v5.0 triangulation is built on
  7. Young-batch censoring fix (DCR-07) — anomaly detection no longer flags young batches as anomalous purely because their late-stage metrics haven't had time to populate. Approach decided by ADR (per-age-bucket norms vs metric-age-eligibility filter). Synthetic test passes.
  8. NULL-vs-zero semantics distinguished (DCR-08) — rate-shaped metrics carry `number | null`, compute layer suppresses on null, anomaly detector ignores nulls. SMS-no-engagement no longer false-flags as zero engagement. (Cross-references BND-01 in Phase 43.)
  9. Polarity consistency audit (DCR-09) — every color-encoded metric display routes through `getPolarity()`. ChartFrame primitive (BND-05) requires `polarity` prop on color-encoded channels.
  10. Apples-and-oranges runtime assertion (DCR-10) — load-bearing UI-gate assumption becomes a verified runtime invariant.
  11. Statistical thresholds documented as ADRs (DCR-11) — `Z_THRESHOLD`, `MIN_GROUPS`, 5% trending, 3-batch baseline, cascade tier breakpoints, `MIN_PLACED_DENOMINATOR_DOLLARS`, account-vs-dollar weighting choice. Each ADR: what it is, alternatives considered, why this one.

**Plans**: _To be scoped via `/gsd:plan-phase` when this phase is ready to start. Likely 3-4 plans: (a) seed-bug fix + per-column aggregation strategy + aggregation contract docs (DCR-01/06), (b) breadth-first metric audit + regression battery (DCR-02/03/04/05), (c) statistical correctness fixes — young-batch censoring, NULL semantics, polarity, runtime invariant (DCR-07/08/09/10), (d) ADR write-up of statistical thresholds (DCR-11)._

### Phase 42: Ingestion-Surface Security Review

**Goal**: Audit the existing security surface (credentials, API routes, client-side exposure, SQL injection) AND produce a forward threat model for v5.0's file-upload + Claude-extraction + stored-scorecard surface so Phase 45 architects ingestion with the threat model already known.
**Depends on**: None within v4.5 (can run in parallel with Phase 41)
**Effort**: Medium (audit + threat model write-up; findings drive Phase 45 architecture)
**Requirements**: SEC-01 through SEC-06

**Success Criteria** (what must be TRUE):
  1. Credential handling audit complete — Snowflake + Anthropic API keys verified server-only, no client-side exposure via sourcemaps, bundles, or network traces
  2. Client-side data exposure audit complete — documented what's reachable via devtools, localStorage, sourcemaps, and the network tab; any unintended exposure (e.g., PII in network traces) is fixed
  3. SQL injection surface review complete — `/api/data`, `/api/accounts`, Metabase import parser, and any NLQ context construction verified against the column allow-list pattern
  4. Forward threat model for v5.0 Phase 45 documented — covers: uploaded file validation (MIME, size, extension), PDF/Excel parsing sandbox, Claude extraction prompt-injection risk, stored scorecard PII handling, rate limits, auth model for uploads
  5. Auth and access control state documented — current model (if any) explicit, v5.0 requirements explicit, gap analysis feeds Phase 45 planning
  6. Dependency security baseline established — `npm audit` clean or triaged, Dependabot or equivalent configured, major-version upgrade paths noted for v5.5 tech-debt sweep

**Plans**: _To be scoped via `/gsd:plan-phase` when this phase is ready to start. Likely 2-3 plans: existing-surface audit, forward threat model for ingestion, dependency + auth baseline. Note: blocked on OAuth landing on Vercel — currently localhost-only; full security review value depends on the deployed surface being real._

### Phase 43: Boundary Hardening

**Goal**: Establish typed parsing boundaries, versioned persistence, Snowflake reliability primitives, a unified `<ChartFrame>` primitive, and tuned caching — so v5.0 inherits a typed substrate, lossless persistence, graceful failure modes, and a single chart system rather than amplifying current debt with three new data shapes.
**Depends on**: Phase 41 partially (DCR-08 NULL semantics is co-implemented with BND-01 canonical row representation — same parser does both). Independent of Phase 42.
**Effort**: Medium-Large (4-5 days; biggest of the v4.5 phases because it touches the most seams)
**Requirements**: BND-01 through BND-06

**Success Criteria** (what must be TRUE):
  1. `BatchRow` and `AccountRow` typed interfaces with resolved types — numbers as `number | null` for rate-shaped fields, branded `BatchAgeMonths`, long-format curves baked in. Parser at `src/lib/data/parse-batch-row.ts`. Static cache loads through the same parser.
  2. Compute layer signature change — every function in `src/lib/computation/` takes the typed row, not `Record<string, unknown>[]`. The three duplicate `coerceAgeMonths` functions collapse to one (the parser).
  3. Persistence layer with `_meta: { schemaVersion, savedAt }` envelope on every blob, migration chain, verified writes (read-back-and-compare), cross-tab `storage` event listener. Replaces direct localStorage in views/columns/lists/presets.
  4. Snowflake reliability — exponential-backoff retry on transient errors, queue-wait separated from query-execution time, request-id correlation (`X-Request-Id` header), sanitized client errors, lightweight circuit-breaker (5 fails / 30s → 60s degraded mode using stale React Query cache).
  5. `<ChartFrame>` primitive — single component owning title, legend slot, tooltip slot, empty/loading/error states, stale-column warning, density-aware padding, polarity prop. All four current charts compose from it. v5.0's triangulation visualizations inherit the frame.
  6. Caching tuned, not disabled — `revalidate: 3600` + tag-invalidation on the data API; React Query `staleTime: 5min`, no refetch on focus; manual Refresh affordance; ADR documenting all three layers.
  7. Snowflake credit consumption drops by ≥10× in the week after deployment (measurable from warehouse usage view) — caching pays for itself.

**Plans**: _To be scoped via `/gsd:plan-phase`. Likely 3 plans: (a) canonical row representation + compute-layer signature change + NULL semantics co-implementation (BND-01/02, with DCR-08), (b) versioned persistence + Snowflake reliability primitives (BND-03/04), (c) ChartFrame primitive + caching tuning (BND-05/06)._

### Phase 44: Vocabulary Lock & Glossary

**Goal**: Lock the canonical domain vocabulary in code and in-product before v5.0 introduces 5 new terms (scorecard, target, triangulation, reconciliation, divergence) on top of 12 existing unowned concepts. Ship a glossary, a `TERMS` registry, a `<Term>` component, resolve the List/View/Workspace conceptual collision, AND surface REVENUE_MODEL as a product dimension once the upstream ETL adds it to `agg_batch_performance_summary` (expected ~2026-05-03).
**Depends on**: VOC-05/06/07 are blocked on ETL adding `REVENUE_MODEL` to `agg_batch_performance_summary`. VOC-01..04 are independent and can run in parallel with Phase 41 / 43. The REVENUE_MODEL work is plumbing-light (~half a day) and gated only on data availability.
**Effort**: Small-Medium for VOC-01..04 (1-2 days). Add ~half a day for VOC-05..07 once REVENUE_MODEL data lands.
**Requirements**: VOC-01 through VOC-07

**Success Criteria** (what must be TRUE):
  1. `docs/GLOSSARY.md` defines every existing domain term in one sentence, written for a new analyst. Cross-references included. Contingency / Debt Sale added once REVENUE_MODEL surfacing lands.
  2. `src/lib/vocabulary.ts` exports `TERMS` registry — every UI label sources from here. Renaming any term is one line.
  3. `<Term name="batch">Batch</Term>` component surfaces the definition on hover/focus from the registry. Applied to KPI card labels, table headers, breadcrumbs, sidebar section labels.
  4. List vs View collision resolved by ADR — choice between (a) merge into Workspaces, (b) make View-contains-List explicit, (c) demote Lists to filters. ADR documents reasoning; UI applies the chosen model.
  5. v5.0 Phase 45 plans extend `TERMS` with the 5 new concepts as part of their plan deliverable; glossary stays current with the milestone.
  6. **REVENUE_MODEL surfaced** (VOC-05) — column added to `ALLOWED_COLUMNS`, `REVENUE_MODEL_VALUES` enum mirrors `ACCOUNT_TYPE_VALUES` shape, exposed via `PartnerListFilters` as a filterable attribute, registered in `TERMS`. Blocked on ETL.
  7. **Canonical scoping decision documented** (VOC-06) — ADR resolves whether REVENUE_MODEL joins (`(partner, account_type, revenue_model)`) or tags. Default: third dimension. Re-evaluated against actual sidebar row counts before implementation.
  8. **UI surfaces the distinction** (VOC-07) — sidebar pair labels, breadcrumbs, and Partner Setup UI reflect the REVENUE_MODEL choice. Mixed-model batch outlier (1/550) handled defensively.

**Plans**: _To be scoped via `/gsd:plan-phase`. Likely 3 plans: (a) glossary + TERMS registry + `<Term>` component (VOC-01/02/03), (b) List/View resolution ADR + UI changes (VOC-04), (c) REVENUE_MODEL surfacing + scoping ADR + UI updates (VOC-05/06/07) — sequenced after ETL completion._

## Key Decisions

| Decision | Rationale |
|----------|-----------|
| v4.5 expanded from 2 to 4 phases (2026-04-26) | Multi-lens audit surfaced two more categories with the same "degrades when deferred" profile: boundary hardening (v5.0 adds 3 new data shapes; without typed parsers and unified chart primitive, debt compounds × 3) and vocabulary lock (v5.0 adds 5 new terms; without a glossary first, conceptual sprawl compounds). Both are pre-v5.0 work by the same logic that put correctness and security pre-v5.0. |
| Phases 43-44 use the reserved slack slots | The 43-44 slots were originally reserved for `/gsd:insert-phase` urgent work during v5.0. Reallocating them to defined v4.5 phases is fine; v5.0 phase numbers (45-49) remain stable. |
| Phase 41 audit is breadth-first, not deep-dive | Goal is to find every correctness bug affecting triangulation, not to rewrite the computation layer. Depth-first rewrites are tech-debt work and belong in v5.5 (DEBT-07..10). |
| Phase 41 statistical fixes are targeted, not architectural | DCR-07 through DCR-11 are surgical fixes to known issues identified in the data review. The architectural rewrite (e.g., Bayesian anomaly detection, t-distribution for small samples) is out of scope. |
| Phase 42 produces a forward threat model | v5.0 Phase 45 architecture is significantly cheaper to get right than to retrofit — ingestion surface (file upload, Claude extraction) has well-known failure modes that should shape design, not patch it later. |
| Phase 42 gated on OAuth landing on Vercel | Currently localhost-only. Full security review value depends on the deployed surface being real — auth model, network exposure, secret handling all change when the app is reachable from the internet. |
| Phase 43 ChartFrame is a primitive, not a full chart-system rebuild | Lift the common shell (legend / tooltip / empty state / polarity) into one component. Each existing chart re-skins to compose from it. The full chart-grammar redesign is a v6.0 design pass. |
| Phase 44 ships before v5.0 begins | v5.0 introduces 5 new domain terms. Locking vocabulary after v5.0 means retrofitting `<Term>` wrappers everywhere. Locking before means v5.0 plans extend `TERMS` as part of their deliverable. |
| Behavioral QA moved to v5.5 | A QA script written before v5.0 ships audits hypothetical flows. Real MBR-prep usage reveals what actually matters. Build the script from observed use, not imagined use. |
| Component decomposition / state consolidation moved to v5.5 | The right seams to extract are the ones v5.0 reveals. Premature decomposition creates abstractions v5.0 then needs to rework. |
