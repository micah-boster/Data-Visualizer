# Milestone v6.0: Proactive Intelligence & Action

**Status:** PLANNED (not started — follows v5.5)
**Phases:** 52-57
**Theme:** "The tool works for you"

## Overview

The tool stops waiting to be opened. It generates briefings, alerts on patterns, feeds directly into MBR deck prep, and connects insights to action in Slack and Notion. v5.0 brought external data in; v6.0 pushes intelligence out.

With triangulated data (internal + scorecard + targets), the proactive layer is 10x more valuable — alerts can say "Partner X is below their contractual target AND their own scorecard shows a different number" instead of just "metric is low."

**Success criteria:** The team gets actionable intelligence without opening the tool. MBR prep starts with a pre-built data package. Every insight has a one-click path to action.

## Phases

### Phase 52: Weekly Partner Highlights

**Goal**: Auto-generated weekly intelligence summary delivered to Notion — key changes, anomalies, target tracking, and recommended focus areas per partner
**Depends on**: v5.0 complete (triangulation data powers the highlights), v5.5 complete (QA baseline)
**Effort**: Medium (scheduling, narrative generation, Notion delivery)
**Requirements**: BRIEF-01 through BRIEF-05
**Status**: FLAGGED FOR REVIEW — may be better served by focusing resources on MBR integration (Phase 54) and pulling weekly highlights as a lighter byproduct of that work. Evaluate after Phase 54 is scoped.

**Success Criteria** (what must be TRUE):
  1. Scheduled weekly run pulls latest data, compares against baselines, generates summary per partner
  2. Highlights structure: key metric changes, new anomalies, target tracking status, scorecard divergence changes
  3. Delivered as a Notion page (one per run, organized in a database)
  4. Slack notification links to the Notion summary
  5. Highlights reference triangulated data (internal + scorecard + target) when available

### Phase 53: Pattern Alerts

**Goal**: Detect multi-signal patterns (not just threshold crossings) and deliver rich, contextual alerts to Slack with suggested actions
**Depends on**: Phase 52 (shared scheduling/intelligence infrastructure) or can be independent if 52 is deprioritized
**Effort**: Medium-Large (pattern library, alert routing, severity tiering, snooze/acknowledge)
**Requirements**: ALERT-01 through ALERT-07

**Success Criteria** (what must be TRUE):
  1. Pattern library includes: consecutive metric decline (3+ periods), acceleration/deceleration, scorecard divergence widening, seasonal anomaly, peer-group outlier emergence
  2. Alerts delivered to Slack with rich context: mini-chart, comparison data, link back to the tool
  3. Severity tiering: informational, attention-needed, urgent — with distinct visual treatment
  4. User can snooze an alert (1 week, 1 month, forever) with optional note
  5. User can acknowledge an alert and optionally create a follow-up action (Notion task)
  6. Alert frequency respects a digest mode: batch multiple low-severity alerts into a daily summary rather than individual messages
  7. Alerts reference triangulated data when available ("Partner X below target AND scorecard divergence growing")

**Plans**: _To be scoped via `/gsd:plan-phase` when this phase is ready to start. Plan count and breakdown depend on codebase state and outcomes from prior phases._

**After Phase 53:** No new phases directly unlocked. Feeds into milestone completion.

### Phase 54: MBR Pipeline Integration

**Goal**: One-click from the partner view to a pre-built MBR data package — auto-stages KPIs, charts, anomalies, target tracking, and scorecard comparison into the existing MBR pipeline skill
**Depends on**: v5.0 complete (triangulation data), existing MBR pipeline skill
**Effort**: Medium-Large (data staging, narrative draft, diff from last MBR, pipeline handoff)
**Requirements**: MBR-01 through MBR-07

**Success Criteria** (what must be TRUE):
  1. "Prep MBR" button on partner detail view triggers data staging
  2. Auto-stages: KPI snapshot, collection curves, anomaly highlights, target tracking status, scorecard comparison
  3. Staged data feeds into the existing MBR pipeline skill with pre-populated context
  4. Claude generates narrative talking points from triangulated data (internal + scorecard + target)
  5. Diff from last MBR: "here's what changed since the last review" with delta values and direction
  6. User can review and edit staged content before triggering deck generation
  7. MBR prep history: track which MBRs were generated, when, and what data was included

**Plans**: _To be scoped via `/gsd:plan-phase` when this phase is ready to start. Plan count and breakdown depend on codebase state and outcomes from prior phases._

### Phase 55: Action Connections

**Goal**: From any data point in the tool, trigger a downstream action — Slack message, Notion task, MBR agenda item — without leaving the app
**Depends on**: Phase 54 (MBR integration establishes the action pattern)
**Effort**: Medium (Slack posting, Notion task creation, action audit trail)
**Requirements**: ACT-01 through ACT-05

**Success Criteria** (what must be TRUE):
  1. "Flag in Slack" on any row/partner/metric: posts a rich message to a selected channel with data context, mini-chart, and link back to the tool
  2. "Create Notion task" on any insight: creates a task in the Partnerships Task Tracker with auto-filled context (partner, metric, current value, target, action suggestion)
  3. "Add to MBR agenda" on any data point: stages the insight for the next MBR prep for that partner
  4. Action audit trail: every action taken from the tool is logged (who, what, when, what data was attached)
  5. Actions are accessible from a right-click context menu on table rows and chart data points

**Plans**: _To be scoped via `/gsd:plan-phase` when this phase is ready to start. Plan count and breakdown depend on codebase state and outcomes from prior phases._

**After Phase 55:** No new phases directly unlocked. Feeds into milestone completion.

### Phase 56: Temporal Intelligence

**Goal**: Time-series reasoning across the tool — vintage comparison, cohort trending, forecasting, and leading indicator discovery
**Depends on**: v5.0 Phase 49 (dynamic curves provide the projection infrastructure)
**Effort**: Large (vintage grouping, cohort overlay, forecasting models, correlation analysis)
**Requirements**: TEMP-01 through TEMP-06

**Success Criteria** (what must be TRUE):
  1. Vintage comparison: "how does this batch compare to the same-age batch from last year?" with side-by-side curve overlay
  2. Cohort trending: group batches by vintage quarter, overlay trajectories on a single chart
  3. Forecasting: "if current trends hold, Partner X finishes the year at $Y" with linear and seasonal-adjusted models
  4. Leading indicator surface: identify which early-batch metrics (months 1-6) correlate most strongly with late-batch outcomes (months 12-24)
  5. Temporal context in Claude NLQ: "when did this trend start?" and "where is this heading?" are answerable
  6. All temporal views respect partner list filtering and saved view state

**Plans**: _To be scoped via `/gsd:plan-phase` when this phase is ready to start. Plan count and breakdown depend on codebase state and outcomes from prior phases._

**After Phase 56:** No new phases directly unlocked. Feeds into milestone completion.

### Phase 57: NLQ Enhancements

**Goal**: Claude NLQ becomes dramatically more powerful with external data, targets, temporal context, and interactive responses
**Depends on**: v5.0 (triangulated data), Phase 56 (temporal intelligence)
**Effort**: Medium (follow-up suggestions, clickable references, query history, multi-source reasoning)
**Requirements**: NLQ-01 through NLQ-05

**Success Criteria** (what must be TRUE):
  1. Follow-up suggestions appear after each response ("You might also want to know...")
  2. Clickable data references in responses: click a number to drill to the source row/partner/metric
  3. Query history persists in Notion (one page per session, organized in a database)
  4. Multi-source queries work: "How does Partner X's self-reported recovery rate compare to our measurement and the contractual target?"
  5. Temporal queries work: "When did Partner X's recovery rate start declining?" and "At current pace, will they hit the Q4 target?"

**Plans**: _To be scoped via `/gsd:plan-phase` when this phase is ready to start. Plan count and breakdown depend on codebase state and outcomes from prior phases._

## Key Decisions

| Decision | Rationale |
|----------|-----------|
| Phase 52 flagged for review | Weekly standalone briefs may be redundant if MBR integration (Phase 54) is deep enough. May be better as a lighter "highlights" byproduct of MBR prep. Evaluate after Phase 54 is scoped. |
| MBR integration is high priority | User deeply integrated MBR prep into workflow. One-click from data to deck is the highest-value action connection. |
| Slack + Notion for actions (not HubSpot yet) | Keep action layer simple in v6.0. HubSpot integration adds complexity — defer to v6.5+ based on usage patterns. |
| Temporal intelligence before NLQ | NLQ enhancements are most powerful when temporal reasoning exists to draw on. Build the data layer first. |
| Pattern alerts include snooze/acknowledge | Alert fatigue is the #1 killer of notification systems. Must have dismissal/management from day one. |
