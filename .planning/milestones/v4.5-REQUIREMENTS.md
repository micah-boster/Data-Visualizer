# Requirements: v4.5 Correctness & Security Foundation

**Defined:** 2026-04-21
**Core Value:** Structural guardrails before v5.0 — verified metrics (so triangulation doesn't amplify silent bugs into partner-relationship damage) and a documented threat model for the ingestion surface (so Phase 45 architects security in, not retrofits it).

## Theme

v4.5 is deliberately narrow. It contains only the work whose value degrades if deferred past v5.0: data correctness on the metrics triangulation will compare, and security review on both the existing surface and the forward threat model for v5.0's file-upload + Claude-extraction pipeline. Behavioral QA and tech-debt sweep are intentionally deferred to v5.5 — those audits are more effective when informed by real post-v5.0 usage.

## Data Correctness Audit (Phase 41)

**Principle:** v5.0 triangulation compares "your numbers vs partner-reported numbers vs target." If "your numbers" have silent bugs, triangulation surfaces them as "the partner disagrees with us" — eroding trust in both the tool and the partner relationship. Every metric on screen must match what a direct Snowflake query returns, at every scope level, before external data lands.

- [ ] **DCR-01**: Summary row aggregation + formatter binding seed bug is fixed. Per-column aggregation strategy (`sum` / `avgWeighted` / `none` / `range`) is declared on each column definition. Footer reuses the column's own formatter (no more Total Accounts rendering as currency, Lender ID summing, or Avg Placed showing NaN). Tracked seed: `.planning/todos/pending/2026-04-19-fix-summary-row-aggregation-and-formatter-binding.md`.
- [ ] **DCR-02**: Every metric displayed in the app (KPI cards, chart series, table cells, summary rows, partner rollups, commitment rate once data arrives) is verified against a direct Snowflake query. Audit produces a metric-by-metric verification table.
- [ ] **DCR-03**: Scope rollups are consistent across drill levels. Selecting a `(partner, product)` pair in the sidebar produces the same KPI values as filtering the root table to that pair and reading its summary row. Root → partner → `(partner, product)` → batch rollups are tested for numerical consistency, not just "looks plausible."
- [ ] **DCR-04**: The Phase 39 `(partner, product)` scope change is specifically audited — no residual partner-level blending survives in any KPI, chart, or aggregated table cell.
- [ ] **DCR-05**: Every correctness bug surfaced by the audit gets a regression fixture or smoke test. Bugs don't just get fixed; they get pinned so they can't silently return.
- [ ] **DCR-06**: Aggregation contract is documented in `docs/` — when a new column is added, the rule for choosing `sum` vs `avgWeighted` vs `none` vs `range` is explicit. Prevents future contributors from re-breaking the summary-row pattern.

## Ingestion-Surface Security Review (Phase 42)

**Principle:** v5.0 Phase 45 introduces a large new security surface — file uploads, Claude extraction, stored scorecard data, potentially PII in contracts. Architecting security in is cheap; retrofitting is expensive and leaves residual risk. v4.5 audits what exists today AND produces the threat model Phase 45 needs.

- [ ] **SEC-01**: Credential handling is verified server-only. Snowflake credentials and Anthropic API keys are not reachable via sourcemaps, client bundles, or network traces. Vercel env var flow is documented end-to-end.
- [ ] **SEC-02**: Client-side data exposure audit is complete. What's reachable via browser devtools, localStorage, sourcemaps, and network inspect is documented. Any unintended exposure (e.g., PII or partner-sensitive fields in network traces) is fixed.
- [ ] **SEC-03**: SQL injection surface review covers `/api/data`, `/api/accounts`, the Metabase import SQL parser, and any Claude NLQ context construction. Each surface is verified against the column allow-list pattern or equivalent parameterization.
- [ ] **SEC-04**: Forward threat model for v5.0 Phase 45 is documented and reviewed. Covers: uploaded file validation (MIME type, size, extension), PDF/Excel parsing sandbox, Claude extraction prompt-injection risk, stored scorecard PII handling, rate limits, upload auth model. Deliverable is a markdown doc Phase 45 planning consumes.
- [ ] **SEC-05**: Auth and access control state is documented. Current model (even if "none, it's a single-user local tool") is explicit. v5.0 requirements are explicit. Gap analysis feeds Phase 45 planning.
- [ ] **SEC-06**: Dependency security baseline established. `npm audit` findings are either clean or triaged with risk notes. Dependabot or equivalent is configured. Major-version upgrade paths (e.g., TanStack v9) are noted for v5.5 DEBT phase intake.
