# Milestone v5.5: Real-Use Hardening

**Status:** PLANNED (not started — follows v5.0, precedes v6.0)
**Phases:** 50-51
**Theme:** "QA informed by usage, not imagination"

## Overview

v5.5 is the post-v5.0 hardening milestone. Its content is the audit work that **improves when deferred**: behavioral QA written from observed MBR-prep workflows rather than hypothetical scripts, and a tech-debt sweep informed by what v5.0 actually added to the codebase.

This is the complement to v4.5. Where v4.5 holds the audits whose value degrades with delay (correctness, security architecture), v5.5 holds the audits whose value grows with observation (behavioral QA, code consolidation). Splitting them by the direction they age is the key decision.

**Success criteria:** `docs/QA-SCRIPT.md` is a living, usage-grounded document the team actually runs before each milestone. The `docs/KNOWN-ISSUES.md` backlog from v4.1 through v5.0 is closed or triaged to v6.0+. Code duplication revealed by v5.0's addition (scorecard normalization, extraction pipelines, triangulation computation) is consolidated against the existing computation layer.

## Phases

### Phase 50: Behavioral QA from Usage

**Goal**: Build `docs/QA-SCRIPT.md` as a living checklist derived from observed MBR-prep workflows and real v5.0 failure modes — not hypothetical flows written before anyone used the feature. Establish regression coverage for v5.0's scorecard ingestion, triangulation views, and dynamic curves.
**Depends on**: v5.0 complete (need actual v5.0 features in use to write meaningful QA)
**Effort**: Medium (observation + checklist authoring + regression battery for v5.0 features)
**Requirements**: QA-01 through QA-06

**Success Criteria** (what must be TRUE):
  1. `docs/QA-SCRIPT.md` covers actual MBR-prep workflows observed during v5.0 use — not hypothetical scenarios. Each entry cites the usage that motivated it.
  2. Regression test battery for v5.0 features: scorecard ingestion paths (PDF, Excel, CSV, email paste), triangulation view rendering under all data-completeness combinations (all 3 sources, missing scorecard, missing target), dynamic curves with various actuals-windows
  3. Keyboard + mouse regression script covers core drill flows end-to-end: root → partner → `(partner, product)` → batch → account, under both single-product and multi-product partners, with filters applied and cleared
  4. Dark/light mode parity check for all v5.0 surfaces (triangulation panels, scorecard library, target management UI, reconciliation dashboard)
  5. Cross-browser smoke test on core flows — Chrome, Safari, Firefox at minimum. Any browser-specific bugs found are fixed or documented with workarounds.
  6. Screen-size parity check — laptop (≤900px height per v4.1 CHT-04), desktop, and any tablet use if relevant

**Plans**: _To be scoped via `/gsd:plan-phase` when this phase is ready to start. Likely 2-3 plans: usage-observation + script authoring, v5.0 feature regression battery, cross-browser/size parity._

### Phase 51: Tech Debt Sweep

**Goal**: Close the `docs/KNOWN-ISSUES.md` backlog accumulated through v4.1/v4.5/v5.0, consolidate code duplication revealed by v5.0's additions (scorecard normalization, extraction pipelines, triangulation computation), and retire dead patterns. Refactor informed by the actual shape v5.0 left the codebase in, not what we guessed before.
**Depends on**: v5.0 complete (tech debt surface isn't knowable until v5.0 lands). Independent of Phase 50.
**Effort**: Medium (scope depends on how much debt v5.0 actually accumulated; triage-first, execute second)
**Requirements**: DEBT-01 through DEBT-06

**Success Criteria** (what must be TRUE):
  1. Every `docs/KNOWN-ISSUES.md` item carried forward from v3.1/v4.0/v4.1/v4.5/v5.0 is either closed or explicitly triaged with a rationale for deferral
  2. Code duplication between v5.0's new computation (scorecard normalization, triangulation math, dynamic-curve projection) and the existing `src/lib/computation/*` layer is consolidated. If scorecards re-implement anything that already existed for internal metrics, they share it.
  3. TanStack v9 migration (KI-13, KI-14) is either done or explicitly scheduled for a later milestone with the upgrade path documented
  4. Major dependency upgrades flagged during v4.5 SEC-06 are executed (or triaged with risk notes if deferred)
  5. Dead code / deprecated patterns / stale abstractions discovered during v4.5 audits and v5.0 development are retired
  6. Hot-path code identified as slow during real v5.0 usage (large scorecard uploads, triangulation with many metrics, dynamic curve recalculation) is profiled and optimized or triaged

**Plans**: _To be scoped via `/gsd:plan-phase` when this phase is ready to start. Likely 2-3 plans: KNOWN-ISSUES backlog closure + dead-code retirement, computation-layer consolidation, dependency upgrades + performance pass._

## Key Decisions

| Decision | Rationale |
|----------|-----------|
| Behavioral QA deferred to v5.5 (not v4.5) | A QA script written before v5.0 ships audits hypothetical flows — real MBR-prep usage reveals what actually matters. Observation-grounded QA is higher-signal per hour invested. |
| Tech-debt sweep deferred to v5.5 (not v4.5) | Every feature shipped adds new debt. Sweeping before v5.0 misses the scorecard/triangulation/dynamic-curve code that the sweep should touch. Sweeping after means one pass handles everything at once. |
| v5.5 is light (2 phases) | Real-use hardening shouldn't block v6.0 for long. Scoped to close the backlog and establish QA discipline, then move on. |
| Phases 50-51 can run in parallel | No dependencies between them. Backlog closure and QA authoring share no artifacts. |
| Computation-layer consolidation in Phase 51 is specific | v5.0 will almost certainly re-implement some existing logic for scorecards — Phase 51 explicitly checks for that and merges. Prevents permanent duplication. |

## Cross-milestone Dependencies

- Depends on v5.0 complete (Phase 45-49 — ingestion, triangulation, reconciliation, dynamic curves must all be in observed use)
- v6.0 Phase 52+ depends on v5.5 only lightly — Phase 50's QA script becomes the regression baseline for v6.0 changes, but v6.0 can start before v5.5 closes if scheduling pressure demands
