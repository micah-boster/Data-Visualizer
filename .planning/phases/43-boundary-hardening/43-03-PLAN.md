```markdown
# ADR 009: Caching Layers (BND-06)

**Date:** 2026-04-30
**Status:** Accepted
**Phase:** 43-boundary-hardening / Plan 03

## Context

The app reads from Snowflake via `/api/data`. Pre-BND-06, the route was `dynamic = 'force-dynamic'` — every page load triggered a fresh Snowflake query. ETL job runs daily; intra-day data is static. Snowflake credit consumption was ~10× what it needs to be.

## Decision

Three caching layers, each with a clear role:

### Layer 1: Next.js route cache (server)
- `/api/data`: `revalidate: 3600` + `unstable_cache(..., ['batch-summary'], { tags: ['batch-data'] })`.
- `/api/revalidate`: POST endpoint with shared-secret auth — daily ETL hits it on completion. NOT called from the client.
- Invalidates: tag-based (`revalidateTag('batch-data')`) on ETL POST; time-based fallback at 1h.

### Layer 2: React Query (client)
- `staleTime: 5 * 60 * 1000` (5 min).
- `gcTime: 30 * 60 * 1000` (30 min).
- `refetchOnWindowFocus: false`.
- Manual `<RefreshButton>` invalidates the `['data']` query — ONLY action. The button does not bypass Layer 1; if Layer 1 is fresh, the React-Query refetch hits the Next cache.

### Layer 3: Snowflake warehouse cache
- Snowflake's own result cache — automatic, identity-of-query keyed.
- Set warehouse `STATEMENT_TIMEOUT_IN_SECONDS` low (already configured).
- Re-runs the same query within 24h hit warehouse cache for free (no compute cost). Layer 1 reduces this hit count further.

## Alternatives considered

- Drop the React Query layer entirely → pushes all freshness logic to server cache + manual refresh. Rejected because optimistic UI and saved-view loading rely on React Query's in-memory cache; redoing them is out of scope.
- Drop the Next route cache → React Query alone handles freshness. Rejected because cold cache means each first-page-load hits Snowflake; multiple users compounding this regresses the credit-consumption goal.
- Have the RefreshButton ALSO hit `/api/revalidate` → rejected. Leaks the revalidate secret to the client (or requires a server proxy that adds complexity for no user benefit). The 1h Layer-1 stale window is acceptable; the daily ETL is the authoritative refresh trigger.
- 4h revalidate window instead of 1h → matches the "daily ETL" cadence more conservatively but means ETL completion → user visibility lag of up to 4h on the first user. 1h is the right balance.

## Why this one
- Daily ETL means data IS static for ~24h; serving cached for 1h is minimal-risk
- Tag-invalidation gives ETL a precise hook
- React Query's 5min staleTime covers user-frequent revisits without thrashing the route cache
- Manual Refresh covers "I just need fresh now" edge case via React-Query bust only

## Partner overrides: NONE

Caching is global. No per-partner tuning.

## When to revisit

- Snowflake credit consumption did NOT drop ≥10× post-deploy — the v4.5 milestone success criterion. Revisit Layer 1 settings.
- ETL job missed the `/api/revalidate` POST and users see 1h-stale data — add a heartbeat / monitor.
- v5.0 introduces user-uploaded scorecards (Phase 45) — ingestion timing differs from daily ETL; per-tag cache keys may diverge.
```
