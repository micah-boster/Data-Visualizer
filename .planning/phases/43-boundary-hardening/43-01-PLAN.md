<objective>
Extend the existing narrow-scope DCR-08 parser at `src/lib/data/parse-batch-row.ts` into the canonical `BatchRow` boundary that BND-01/BND-02 require. After this plan, every function in `src/lib/computation/` consumes typed `BatchRow[]` — the bag-of-strings (`Record<string, unknown>[]`) does not appear inside compute. Static cache JSON loads through the same parser, eliminating days-vs-months ambiguity by construction (branded `BatchAgeMonths`). The three duplicate `coerceAgeMonths` definitions collapse to one. DCR-08's rate-shaped `number | null` contract widens from "narrow scope" (7 fields, current state) to "every typed numeric field" via the row interface.

Purpose: v5.0 introduces 3 new data shapes (scorecards, targets, triangulation rows). If they each get parsed ad-hoc the way batch rows currently are, v5.0 inherits 3× the representation debt. This plan establishes the boundary-discipline pattern on the existing batch-row surface so v5.0 inherits it as a precedent rather than re-deriving it.

Output: `BatchRow` / `AccountRow` interfaces in `src/lib/data/types.ts`, an extended `parseBatchRow` / `parseBatchRows` that produces them, full-row consumption inside compute, and a smoke test pinning the filter+warn drop semantics + branded-age construction + curve reshape.
</objective>

Create `src/lib/data/types.ts` exporting the canonical typed-row contract that BND-02 will consume. This is the contract file — implementation comes in Task 2.

Required exports:
- `export type BatchAgeMonths = number & { readonly __brand: 'months' }` — branded number type. Resolves the days-vs-months ambiguity by construction.
- `export function asBatchAgeMonths(n: number): BatchAgeMonths` — single coercion site (replaces the three duplicate `coerceAgeMonths` definitions in compute). Throws `RangeError` on negative or non-finite input; callers in the parser handle the throw and feed it into the filter-and-warn drop pipeline.
- `export interface CurvePoint { month: number; recoveryRate: number }` — long-format curve point. Mirrors the shape already used by `BatchCurve` consumers downstream (see `compute-cross-partner.ts` and `collection-curve-chart.tsx`).
- `export interface BatchRow` — typed batch-summary row. Required fields:
  - Identity: `partnerName: string`, `accountType: string`, `batchName: string`, `lenderId: string`
  - Age: `batchAgeMonths: BatchAgeMonths`
  - Volume: `totalAccounts: number`, `totalAmountPlaced: number`, `totalConvertedAccounts: number`, `totalCollectedLifeTime: number`, `avgAmountPlaced: number`
  - Horizon collection metrics: `collectionAfter3Month: number | null`, `collectionAfter6Month: number | null`, `collectionAfter12Month: number | null` (null when batch is too young per DCR-07's eligibility gate)
  - Rate-shaped engagement (DCR-08, already enforced by current parser): `smsOpenRate: number | null`, `smsClickRate: number | null`, `emailOpenRate: number | null`, `emailClickRate: number | null`, `callConnectRate: number | null`, `callRpcRate: number | null`, `disputeRate: number | null`
  - Long-format curve: `curve: CurvePoint[]` (built once at parse time from the wide `COLLECTION_AFTER_*_MONTH` columns; replaces the per-render `reshape-curves.ts` work for downstream BND-02 consumers — `reshape-curves.ts` itself stays for any callsite that still wants the helper, but the pre-baked `curve` field is the canonical access path)
  - Raw passthrough: `raw: Record<string, unknown>` — the original Snowflake row for any consumer not yet migrated to the typed shape (sidebar + table currently access `row.PARTNER_NAME` directly via `getValue('PARTNER_NAME')`; preserving raw lets us migrate compute first without breaking those surfaces)
- `export interface AccountRow` — minimal canonical interface for `/api/accounts` rows. Only the fields the app actually reads today (LOAN_ID, PARTNER_NAME, BATCH_, ACCOUNT_TYPE, status fields). Same `raw` passthrough.
- `export interface ParseBatchRowsResult { rows: BatchRow[]; dropped: Array<{ reason: string; sample: Record<string, unknown> }> }` — return shape for the batch parser, threading the filter-and-warn metadata to callers.

Document the days-vs-months ambiguity resolution in the JSDoc on `BatchAgeMonths` — Snowflake column is `BATCH_AGE_IN_MONTHS` (months), but several pre-Phase-41 sites coerced to days via `* 30` or `/ 30` rounding errors. The brand makes the unit visible at every callsite.

DO NOT modify any consumer yet — this task is pure type contract.
