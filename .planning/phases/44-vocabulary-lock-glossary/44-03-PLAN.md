Use the same Markdown format as ADR 0001 (Status / Context / Decision / Alternatives / Consequences). Frontmatter:

- **Status:** Accepted
- **Date:** {ISO date when this plan runs}

- **Requirement:** VOC-06 (also load-bearing for VOC-05, VOC-07)

**Sections:**

- **Status:** Accepted.
- **Context:** v4.1 Phase 39 established `(partner, product)` as the canonical unit of analysis to enforce the apples-and-oranges rule (Contingency vs Debt Sale economics differ enough that aggregating them produces meaningless numbers). The ETL has now added `REVENUE_MODEL` to `agg_batch_performance_summary`, surfacing the third axis of the apples-and-oranges rule. We need to decide how it integrates into the existing scoping model.
- **Decision:** REVENUE_MODEL joins the unit-of-analysis as a third dimension. Unit becomes `(partner, product, revenue_model)`. KPIs, collection curves, and anomaly detection treat Contingency and Debt Sale as different scopes and never blend them. The sidebar splits a partner-product pair into multiple rows when that partner carries multiple revenue models for the same product (UI work in Plan 44-04).
- **Alternatives Considered:**
  - **Treat REVENUE_MODEL as a tag/filter on the existing pair (no third dimension)** — Rejected: tagging implies aggregations CAN safely cross the dimension; the apples-and-oranges rule explicitly forbids this. Would force every consumer (KPIs, curves, anomaly compute) to add their own per-revenue-model split logic, defeating the structural-enforcement-at-the-selection-layer pattern Phase 39 established.
  - **Make REVENUE_MODEL a column-only filter (no scoping role)** — Rejected: same problem as tagging, plus it leaves users unable to easily see Happy Money 3P-Contingency separately from Happy Money 3P-DebtSale without manual filter dance.
  - **Re-evaluate based on actual sidebar row count after ETL lands** — CONTEXT.md flagged this as a check. Run the row-count audit during this plan's execution: count distinct (partner, product, revenue_model) tuples in the dataset. If the count is unwieldy (rough threshold: > 50 rows total or > 5 rows per partner), reopen the decision before commit. Otherwise proceed with third-dimension scoping. Document the actual count in this ADR's Consequences section.
- **Consequences:**
  - Sidebar pair count grows by however many distinct revenue models exist within multi-model (partner, product) combos. Affirm contingency-only stays one row; Happy Money may split into 2 (1P-Contingency, 1P-DebtSale) or 4 rows depending on actual data shape.
  - All compute-layer code that operates on `BatchRow[]` filtered by `(partner, product)` must additionally filter by `revenueModel`. Phase 43 (Boundary Hardening) will codify this in the typed parser; for now, partition-by-pair logic in `usePartnerStats` and friends gains a third predicate.
  - Saved Views and Partner Lists get an additional `REVENUE_MODEL` filter axis. Lists pre-Phase-44 parse cleanly (additive `.optional()` evolution).
  - The 1/550 mixed-revenue-model batch outlier (data-quality issue from the 549/550 audit) displays under its dominant revenue model with a small warning chip in Plan 44-04. Future ETL fix may eliminate this case entirely.
  - **Records actual sidebar row count at adoption:** "**Sidebar pair count after REVENUE_MODEL split:** {N} rows (was {M} pre-split). Distribution: {brief — e.g. '4 partners single-row, 2 partners split into 2 rows each, 1 partner split into 3 rows'}." This is recorded at execution time by reading the production dataset. If N > 50 or any partner produces > 5 rows, pause and notify the user before commit.
