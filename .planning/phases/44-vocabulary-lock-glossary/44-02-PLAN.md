**Sections to include:**

- **Status:** Accepted.
- **Context:** Today the sidebar renders Partner Lists and Saved Views at the same visual hierarchy level with no signpost about how they relate. A new user reading the sidebar can't tell that a View is a UI snapshot (columns + sort + filters + drill scope) while a List is a data filter on partners. v5.0 introduces three additional concepts (scorecards, targets, triangulation surfaces); without resolving this collision first, the sidebar IA compounds.
- **Decision:** View-contains-List explicit hierarchy. Keep both terms. A View is a UI snapshot; a List is a data filter that a View can contain via the optional `listId` reference. The sidebar surfaces this by nesting any List a View carries underneath that View. The Partner Lists section continues to render standalone, independently-managed Lists for cases where the user wants the data filter without saving the surrounding UI state.
- **Alternatives Considered:**
  - **Merge into a single "Workspace" concept** — would force a global rename across the codebase + UI + saved-state migrations + user mental-model retraining. Rejected: high churn for low marginal clarity. Two terms with a clear hierarchy is more precise than one term that has to absorb both meanings.
  - **Demote Lists to filters (kill the Lists concept)** — Lists carry persistence semantics Views don't (a List is refreshable when its source attributes change, e.g. PRODUCT_TYPE expanding). Demoting them loses that affordance and forces users to recreate the filter every time. Rejected.
  - **Keep both flat, add a section divider** — minimal change, but doesn't actually convey the hierarchy. The reader still has to learn the difference; the divider just adds visual noise. Rejected.
- **Consequences:**
  - `ViewSnapshot` carries an additive `listId?: string` field (introduced in this plan). Legacy views without the field continue to parse cleanly via zod's `.optional()` semantics — Phase 39 schema-evolution pattern, no migration.
  - Sidebar Views group renders nested rows when a View carries a List (Task 2 wires this).
  - The TERMS registry (Plan 44-01) defines `view` and `list` separately — the ADR is the canonical source for what each one IS; the registry is the surface that exposes those definitions to users.
  - v5.0 Phase 45+ work that adds scorecard / target / triangulation surfaces inherits this pattern: surfaces nest under the conceptual parent, not at peer level.
  - No "Workspace" rename — today's mental model preserved.
  - **Schema state at ADR adoption (Phase 44):** ViewSnapshot {Finding A: had | Finding B: did not have, this plan added | Finding C: had as dead code, this plan adopts} an explicit `listId` field. This ADR governs the conceptual model and the sidebar render layer enforces it via the `listId` lookup.

**Avoid:**
- Adding ADRs 0002+ in this plan (those land in Plan 44-03 for REVENUE_MODEL scoping). One ADR per plan.
- Fancy frontmatter beyond the lightweight metadata block. Heavy YAML / templating slows down the writing-an-ADR habit.
- Predicting the v5.0 sidebar layout in concrete detail — the ADR records the principle ("surfaces nest under the conceptual parent"), not specific v5.0 nesting choices.
- Adding any UI that lets the user EDIT `view.listId` from inside the app — out of scope. Existing flow that creates a View can later be extended to attach a List via a follow-up plan; this plan ships the schema and the render path.
- Bumping schemaVersion — additive `.optional()` is forward-compatible by definition.
