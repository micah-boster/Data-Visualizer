SQL (use the CONFIRM.md decisions from Task 0 — default shape if probes fell back):
```sql
WITH latest AS (
  SELECT LENDER_ID, BATCH_, PRICING_TYPE, MAX(VERSION) AS version
  FROM BOUNCE.FINANCE.CURVES_RESULTS
  WHERE PROJECTED_FRACTIONAL IS NOT NULL
  GROUP BY LENDER_ID, BATCH_, PRICING_TYPE
), deduped AS (
  SELECT
    c.LENDER_ID, c.BATCH_, c.COLLECTION_MONTH, c.PROJECTED_FRACTIONAL,
    ROW_NUMBER() OVER (PARTITION BY c.LENDER_ID, c.BATCH_, c.COLLECTION_MONTH ORDER BY c.VERSION DESC) AS rn
  FROM BOUNCE.FINANCE.CURVES_RESULTS c
  JOIN latest l
    ON c.LENDER_ID = l.LENDER_ID
   AND c.BATCH_ = l.BATCH_
   AND c.PRICING_TYPE = l.PRICING_TYPE
   AND c.VERSION = l.version
  WHERE c.PROJECTED_FRACTIONAL IS NOT NULL
)
SELECT
  LENDER_ID AS "lenderId",
  BATCH_ AS "batchName",
  COLLECTION_MONTH AS "month",
  PROJECTED_FRACTIONAL * 100 AS "projectedRate"   -- per CONFIRM.md units decision; drop the *100 if CONFIRM.md says percentage already
FROM deduped
WHERE rn = 1
ORDER BY LENDER_ID, BATCH_, COLLECTION_MONTH
```

```typescript
'use client';
import { useMemo } from 'react';
import { useQuery } from '@tanstack/react-query';
import type { CurvePoint } from '@/types/partner-stats';
import type { CurvesResultsResponse } from '@/types/curves-results';

export function useCurvesResults() {
  return useQuery<CurvesResultsResponse>({
    queryKey: ['curves-results'],
    queryFn: async () => {
      const res = await fetch('/api/curves-results');
      if (!res.ok) {
        const err = await res.json().catch(() => ({ error: 'Failed to fetch projections' }));
        throw new Error(err.error ?? 'Failed to fetch projections');
      }
      return res.json();
    },
    staleTime: 5 * 60 * 1000,  // projection data refreshes on ETL cadence
  });
}

/** Map<"lenderId||batchName", CurvePoint[]> — stable across renders, rebuilt only when data changes. */
export function useCurvesResultsIndex(): Map<string, CurvePoint[]> {
  const { data } = useCurvesResults();
  return useMemo(() => {
    const map = new Map<string, CurvePoint[]>();
    if (!data?.data) return map;
    for (const row of data.data) {
      const key = `${row.lenderId}||${row.batchName}`;
      const pts = map.get(key) ?? [];
      // Phase 40 v1 is rate-only; amount = 0 per Deferred Ideas ($ trajectory out of scope).
      pts.push({ month: row.month, amount: 0, recoveryRate: row.projectedRate });
      map.set(key, pts);
    }
    return map;
  }, [data?.data]);
}
```

Why not a single inlined JOIN into /api/data: research Pitfall + "alternatives considered" — adding 80 months × ~500 batches to the base query would 80× response size. Sibling query isolates latency and lets chart stay functional without projections.

Extend `src/hooks/use-partner-stats.ts` to merge projections. The hook currently accepts `(partnerName, allRows)` and returns `PartnerStats | null`. We need to:
1. Pull the projection index via `useCurvesResultsIndex()` at the top of the hook (outside the memo).
2. Build a `lenderByBatch: Map<string, string>` from `partnerRows` so each batch resolves to its own `LENDER_ID` — per Pitfall 1, do NOT assume partner-level uniformity. Walk `partnerRows` once, read `String(r.LENDER_ID ?? '')` alongside `String(r.BATCH_ ?? '')`.
3. After `reshapeCurves(partnerRows)`, map each curve: `{ ...curve, projection: projectionIndex.get(`${lenderByBatch.get(curve.batchName) ?? ''}||${curve.batchName}`) }`. If the lookup returns `undefined`, `projection` stays undefined (batch has no modeled data — graceful degradation).
4. Include `projectionIndex` and `lenderByBatch` in the memo's dependency array. `lenderByBatch` is derived from `allRows`, so it can be computed INSIDE the memo alongside the filter; `projectionIndex` comes from the hook and is already memo-stable across unrelated renders.

The final hook shape (showing added lines; rest unchanged):
```typescript
export function usePartnerStats(
  partnerName: string | null,
  allRows: Record<string, unknown>[],
): PartnerStats | null {
  const projectionIndex = useCurvesResultsIndex();  // NEW — stable Map
  return useMemo(() => {
    if (!partnerName || allRows.length === 0) return null;
    const partnerRows = allRows.filter((r) => String(r.PARTNER_NAME ?? '') === partnerName);
    if (partnerRows.length === 0) return null;

    // NEW: per-batch lenderId lookup (Pitfall 1 — partner may span multiple lenders)
    const lenderByBatch = new Map<string, string>();
    for (const r of partnerRows) {
      const batch = String(r.BATCH_ ?? '');
      const lender = String(r.LENDER_ID ?? '');
      if (batch && lender && !lenderByBatch.has(batch)) lenderByBatch.set(batch, lender);
    }

    const norms = computeNorms(partnerRows);
    const anomalies = partnerRows.length >= 3 ? computeAnomalies(partnerRows, norms) : undefined;

    // NEW: merge projection onto each curve (graceful: stays undefined when no coverage)
    const curves = reshapeCurves(partnerRows).map((curve) => ({
      ...curve,
      projection: projectionIndex.get(
        `${lenderByBatch.get(curve.batchName) ?? ''}||${curve.batchName}`,
      ),
    }));

    return {
      kpis: computeKpis(partnerRows),
      norms,
      curves,
      trending: computeTrending(partnerRows),
      anomalies,
    };
  }, [partnerName, allRows, projectionIndex]);
}
```

DO NOT modify `reshapeCurves` — keeping the merge at the hook layer preserves `reshapeCurves`'s single responsibility (raw row → batch curve). Phase 40 is a composition step, not a reshape step.

DO NOT add loading/error handling for the curves-results query here. Chart must render actuals immediately while projection loads (per RESEARCH: "don't block first paint"). Absent `projection` is always a valid state.
