CONTEXT lock: "Cross-fade with ~150ms overlap. Skeleton fades out while content fades in, brief overlap window." Incoming content uses --ease-decelerate (arrival curve), outgoing skeleton uses --ease-default. CONTEXT locks this at data-display.tsx loading boundaries — the main consumer of skeletons in the drill region.

**Step 1 — Identify skeleton render boundaries.** Search data-display.tsx for:
- `<Skeleton />` (from src/components/ui/skeleton.tsx)
- Dynamic import loaders at lines 39-97 (per RESEARCH) — these may render a Skeleton fallback via `<Suspense fallback={<Skeleton />}>`
- Any `loading ? <Skeleton /> : <Content />` conditional

Typical pattern:
```tsx
{loading ? <SkeletonVariant /> : <Content data={data} />}
```

**Step 2 — Apply the 150ms overlap cross-fade.** Two implementation options:

**Option A — React dual-mount with timed opacity (recommended):**

Track both the loading state AND a delayed "skeleton finished fading out" state:

```tsx
const [skeletonVisible, setSkeletonVisible] = useState(loading);
const [contentReady, setContentReady] = useState(!loading);

useEffect(() => {
  if (loading) {
    setSkeletonVisible(true);
    setContentReady(false);
  } else {
    // Data arrived — start 150ms overlap window
    setContentReady(true);
    const t = setTimeout(() => setSkeletonVisible(false), 150);
    return () => clearTimeout(t);
  }
}, [loading]);

// Render both during the overlap window
return (
  <div className="relative">
    {skeletonVisible && (
      <div className={cn(
        'transition-opacity duration-quick ease-default',
        contentReady ? 'opacity-0' : 'opacity-100',
        contentReady && 'absolute inset-0', // absolute-position once content renders
      )}>
        <SkeletonVariant />
      </div>
    )}
    {contentReady && (
      <div className={cn(
        'transition-opacity duration-normal ease-decelerate',
        contentReady ? 'opacity-100' : 'opacity-0',
      )}>
        <Content data={data} />
      </div>
    )}
  </div>
);
```

Easing assignment per CONTEXT:
- Skeleton fading out: `--ease-default` (state change) + `duration-quick` (120ms, quick exit)
- Content fading in: `--ease-decelerate` (arrival) + `duration-normal` (200ms, proper entrance)

The 150ms overlap is the window where both are visible together. Timing: content starts fading in at t=0 (ease-decelerate 200ms), skeleton starts fading out at t=0 (ease-default 120ms), skeleton unmount at t=150ms. Overlap window = 0-150ms where both are partially visible.

**Option B — Simpler re-key fade (if dual-mount is too invasive):**

If the data-display.tsx loading boundary is already inside a wrapper that re-keys on data, use a single `transition-opacity duration-normal ease-decelerate` on the wrapper and accept the slight flash of no-content between skeleton unmount and content mount. Worse UX than Option A but ships faster. Prefer Option A unless data-display's render structure makes the dual-mount impractical (e.g., skeletons are buried in distant child components — in which case this DS-27 implementation needs a pattern abstraction that's out of scope for this plan).

**Step 3 — Optional: abstract into a `<FadeSwap loading={...}>` pattern.** If data-display.tsx has 3+ skeleton sites, extract the dual-mount logic into a reusable component at `src/components/patterns/fade-swap.tsx`. This is a Claude-discretion call:
- If ≤2 skeleton sites: inline the logic at each, keep this plan scoped
- If 3+ sites: create FadeSwap pattern, migrate sites

Start by inspecting the file; decide. Document in SUMMARY. Prefer inline for this plan — FadeSwap extraction can follow in a later plan if needed.

**Step 4 — Guard check:motion.** All utilities used (duration-quick, duration-normal, ease-default, ease-decelerate) are token-backed. The `opacity-0`/`opacity-100` utilities are fine. `absolute inset-0` positioning is layout, not motion — not flagged.

**Step 5 — Verify reduced-motion path.** With the global override from Plan 30-01, `transition-duration: 0ms !important` forces instant skeleton-to-content swap under OS reduce-motion. The dual-mount timer still fires at 150ms, so the skeleton still unmounts on schedule — just without animation. Document in SUMMARY that this is the expected behavior (RESEARCH Open Q#3 resolution).

**Step 6 — animate-pulse during fade-out.** The skeleton is still `animate-pulse`-animating while opacity fades. The global reduced-motion override disables animate-pulse there. In normal mode, the pulsing skeleton fades to opacity 0 — visually acceptable (disappearing skeleton in its natural rhythm). No special handling needed.
