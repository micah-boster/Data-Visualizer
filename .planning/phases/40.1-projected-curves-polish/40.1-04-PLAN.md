function formatAggregate(value: number | null, type: string, label: string): string {
  if (value == null) return '\u2014';
  let formatted: string;
  switch (type) {
    case 'currency': formatted = formatCurrency(value); break;
    case 'percentage': formatted = formatPercentage(value); break; // ⬅ ×100 happens here
    case 'count': formatted = formatCount(value); break;
    default: formatted = formatNumber(value); break;

  return `${label}: ${formatted}`;

**Critical observation:** `TableFooter` does NOT consult `column.columnDef.footer`. It computes display from `aggregates[column.id]` + `meta.type`. So a `footer: () => 'Avg: 42.3%'` callback on the ColumnDef alone won't be picked up — the executor MUST also extend `TableFooter` to honor the per-column override.

**Two viable paths (executor picks A unless they discover an issue during implementation, then falls back to B):**

A. **Add `meta.footerFormatter` on the ColumnDef.** Type signature: `footerFormatter?: (avg: number | null, label: string) => string`. `TableFooter` checks `meta.footerFormatter` first; if present, uses its output (with `agg.primary` and `agg.label`); else falls through to the existing `formatAggregate(agg.primary, colType, agg.label)`. Surgical, type-safe, narrow scope.

B. **Per-column `meta.type` override.** Add new `meta.type` values `'percentage-100'` (already-on-0..100 scale) and `'percentage-points'` (signed, 'pp' suffix), extend `formatAggregate` switch. Slightly larger blast radius (touches the formatter switch + introduces type values that other consumers might also want to consume). VERIFICATION.md flagged this as "exceeds v1 polish scope" — only fall back to this if Path A produces a footer-rendering surprise.

**Recommended:** Path A. Smallest possible change, doesn't introduce new meta.type values, and the `footerFormatter` extension point becomes a reusable escape hatch for any future column that needs a non-standard aggregate format.

**Class breakdown:**
- `line-clamp-2` — wraps to up to 2 lines, then ellipsis (Tailwind utility; uses `-webkit-line-clamp` under the hood, supported in all evergreen browsers).
- `break-words` — allows long unbreakable strings (e.g., a hypothetical "Modeled12moAvgRecoveryRate" merged label) to break mid-word as a last resort. Without this, an over-long single word can overflow the column.
- `leading-snug` — tightens line-height to `1.375` so two lines feel cohesive (Tailwind default `leading-normal` = `1.5` would create more vertical air than the header chrome can absorb).
- `min-w-0` — preserved from the original; required for the flex parent's `truncate`/wrap mechanics to honor the column's `size`.
- `max-w-[180px]` — REMOVED. line-clamp-2 honors the column's actual width (set per-col via ColumnDef.size); imposing a 180px cap would constrain longer columns (e.g., a 240px-wide identity column would still get 180px-wrapped text). Per-column size is the right constraint.
