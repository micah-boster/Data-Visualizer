### Locked Decisions

#### Chart visibility scoping

| Context | Phase 40 (current) | Phase 40.1 |
|---------|--------------------|-----------|
| Batch-level drill (`drillState.level === 'batch'`) | Projection rendered | Projection rendered, **default-on** (no change) |
| Partner-aggregate, multiple batches visible | All projections rendered | **No projections rendered** |
| Partner-aggregate, **single batch visible** (legend-narrowed to one) | All projections rendered | Projection rendered for that one batch |

**Trigger rule:** render projection `<Line>`s when `drillState.level === 'batch'` OR `visibleBatchCount === 1`. Otherwise hide all projection lines.

**Selection mechanism:** "selected" = legend-narrowed-to-one. No new pin/highlight interaction in v1; the legend already drives visibility and reusing it keeps the mental model coherent. If "narrow to one" feels constraining later we can add a click-to-pin layer (see Deferred).

#### Investigate: batch-level coverage gaps

Live testing surfaced "some batches weren't graphing" at batch-level view. Phase 40 includes a graceful-degradation contract: batches without `PROJECTED_FRACTIONAL` rows in `CURVES_RESULTS` render actuals only (no phantom line). That is correct behavior for genuinely-uncovered batches — but the absence is invisible: the user has to NOTICE the dashed line is missing.

**Plan tasks must:**
1. Query coverage: for each batch in `/api/data`, is there a matching `(LENDER_ID, BATCH_)` row in `/api/curves-results`? Tabulate matched / unmatched.
2. Audit Pitfall 1 paths in `use-partner-stats.ts:55-65` — the `lenderByBatch` map uses first-seen-wins; if a partner has cross-lender batch-name collisions, the second is misrouted. Likely rare, but verify.
3. Audit Pitfall 1b — multi-lender partners (e.g. `bounce_imprint` + `bounce_imprint_dp` + `bounce_imprint_dsc`): each batch carries its own correct lender, so the map should be correct, but worth reading partnerRows once for an Imprint drill to confirm the merge produces non-null projections for batches that have warehouse coverage.
4. **Add an "absent coverage" indicator at batch-level view** — when projection is null on a batch drill, show a small caption (e.g. `No modeled curve for this batch`) somewhere visible (top-right of chart? KPI baseline absent state already does this for KPIs). Same idea for "your selected batch has no modeled coverage" in the aggregate-narrowed-to-one case.

#### Table integration

**What's added:** for each visible `COLLECTION_AFTER_X_MONTH` milestone column (currently 6mo, 12mo at root), add two sibling columns:

- `Modeled X mo` — cumulative modeled %, formatted as percentage
- `Δ vs Modeled X mo` — actual − modeled, signed, polarity-colored (green when actual exceeds modeled, red below)

**Source:** the same `BatchCurve.projection` already merged onto each row by `usePartnerStats`. Per-row lookup by `(LENDER_ID, BATCH)` already exists; just expose the cumulative value at the requested month onto the table data shape.

**Visibility gating:** the panel-level `BaselineSelector` from Phase 40 is the single source of truth. Switching to "Modeled curve" reveals modeled + delta columns AND switches KPI deltas AND enables the chart projection-on-narrowed-batch behavior described above. Switching back to "Rolling avg" reverses all three. One mental model: "modeled mode" affects chart + KPIs + table together.

**Persistence:** the selected baseline mode persists across sessions via `localStorage` (key: `gsd:baselineMode`, default `'rolling'`). Phase 40 always reset to rolling on load — that creates friction once the table integration lands, since users staying in modeled-mode would have to re-toggle every session. First load with no stored value → default `'rolling'` (zero-regression for users who haven't seen the toggle).

**Aggregate-row scope:** modeled columns appear only at scopes where rows resolve to single batches. At root (cross-partner rollup) and partner-aggregate (multi-batch rollup) levels, the modeled + delta columns are hidden even in modeled mode — modeled is a per-batch concept, and synthesizing an aggregate is exactly the lossy operation Phase 40's CONTEXT rejected.

## Summary

Phase 40.1 is a **pure consumer-side polish** on Phase 40. Zero data-pipeline change: no new API route, no new hook, no SQL edits, no `BatchCurve` shape change. Every signal needed is already on the wire (`BatchCurve.projection?: CurvePoint[]` per batch, `partnerStats.curves`, the `BaselineSelector` + `baselineMode` state, the KPI-side baseline-absent recipe, the per-row table data flowing through `definitions.ts`).

Five surgical changes:

1. **Chart visibility gate** — replace the per-line `hide={!isVisible}` predicate at `collection-curve-chart.tsx:454` (projection `<Line>`) with `hide={!isVisible || (drillState.level !== 'batch' && visibleBatchKeys.length !== 1)}`. The `visibleBatchKeys` array is already in scope; `drillState` needs to be threaded in (currently the chart doesn't see drill level — see the Pitfall section).
2. **Table modeled columns** — extend the table's column flow with derived columns. The cleanest seam is `definitions.ts` (which builds the partner-level `columnDefs` from `COLUMN_CONFIGS`) — append `MODELED_AFTER_6_MONTH`, `DELTA_VS_MODELED_6_MONTH`, `MODELED_AFTER_12_MONTH`, `DELTA_VS_MODELED_12_MONTH` after their actual siblings. Each is a `cell:`-rendered column reading `BatchCurve.projection` via the table `meta` channel (where `trending`, `norms` already travel).
3. **localStorage persistence** — change the `useState<BaselineMode>('rolling')` initializer at `data-display.tsx:283` to a lazy initializer that reads `localStorage.getItem('gsd:baselineMode')`. Wrap `setBaselineMode` calls so writes also persist. Pattern lifted verbatim from `use-heatmap-preference.ts`.
4. **Coverage-absent caption** — small `text-caption text-muted-foreground` rendered above or beside the chart when the gate "should render projection" is true but the focused batch has `projection === undefined`. Mirrors the KPI baseline-absent caption shipped in Phase 40-03.
5. **BaselineSelector unification** — `data-display.tsx` already owns `baselineMode` state and threads it to KPI cards; the same prop needs to flow into (a) the table (via `tableOptions.meta` carrying both `baselineMode` and a `projectionByBatch` lookup map) and (b) the chart (via `baselineMode` prop or context). Both extensions follow the existing `partnerStats.curves` plumbing.

The work is small, well-bounded, and has no architectural ambiguity. Risks are limited to: (a) the chart not currently seeing `drillState.level` (cheap to thread), (b) ensuring the new derived columns flow into CSV export without changes (analysis below — they will, with a caveat about how `formattedValue` is captured for export), and (c) the small "where exactly does the absent-coverage caption mount" decision (DataPanel header vs. an overlay).

**Primary recommendation:** Add a single Wave-of-tasks plan: (1) thread `drillState.level` + `visibleBatchKeys` count into the chart's projection-hide predicate plus mount the coverage-absent caption; (2) extend `definitions.ts` with derived modeled + delta columns gated on `meta.baselineMode === 'modeled'`; (3) extract a `useBaselineMode()` hook (mirrors `useHeatmapPreference()`) to handle localStorage persistence; (4) thread `baselineMode` + a per-batch `projectionIndex` into the table's `meta` and into `CollectionCurveChart`; (5) document the multi-lender batch-name-collision audit as an investigation task with a decision gate (defensive fix vs. audit-only confirmation). Tests: existing smoke pattern (Phase 40 already has `compute-projection.smoke.ts`, `pivot-curve-data.smoke.ts`) — extend with cell-formatter tests for the signed-delta cell.

### Pattern 3: localStorage persistence for `baselineMode`

**What:** Replace the inline `useState<BaselineMode>('rolling')` at `data-display.tsx:283` with a custom hook that reads/writes to `localStorage` under key `gsd:baselineMode`. Mirror `use-heatmap-preference.ts` exactly.

**When to use:** Always. Required for PRJ-09.

**Example:**
```typescript
// Source: NEW src/hooks/use-baseline-mode.ts — verbatim mirror of
// src/hooks/use-heatmap-preference.ts with payload validation.
'use client';

import { useState, useCallback } from 'react';
import type { BaselineMode } from '@/components/kpi/baseline-selector';

const STORAGE_KEY = 'gsd:baselineMode';

function isValid(v: unknown): v is BaselineMode {
  return v === 'rolling' || v === 'modeled';
}

/**
 * Manages baseline-mode preference persisted in localStorage.
 *
 * Defaults to 'rolling' when no stored value exists OR when the stored value
 * is corrupt (zero-regression for first-load and graceful degradation for
 * unexpected payloads). SSR-safe.
 *
 * The reset-to-rolling effect at data-display.tsx (when modeledAvailable
 * flips false) STILL APPLIES — persistence does not override the safety
 * mechanism. If a user has 'modeled' persisted but lands on a scope without
 * coverage, the effect flips them back to 'rolling' for that session.
 */
export function useBaselineMode(): [BaselineMode, (next: BaselineMode) => void] {
  const [mode, setModeState] = useState<BaselineMode>(() => {
    if (typeof window === 'undefined') return 'rolling';
    try {
      const stored = localStorage.getItem(STORAGE_KEY);
      return isValid(stored) ? stored : 'rolling';
    } catch {
      return 'rolling';
    }
  });

  const setMode = useCallback((next: BaselineMode) => {
    setModeState(next);
    try {
      if (typeof window !== 'undefined') {
        localStorage.setItem(STORAGE_KEY, next);
      }
    } catch {
      // Quota / private-browsing — silently skip persistence.
    }
  }, []);

  return [mode, setMode];
}
```

**Wiring at the consumer:**
```tsx
// Source: replaces data-display.tsx:283

```

The reset-to-rolling effect at `data-display.tsx:298-306` MUST keep working — when `modeledAvailable` becomes false the call `setBaselineMode('rolling')` should also persist. Since `setBaselineMode` is now the persisting setter, this happens automatically.

### Pattern 4: Virtual modeled-and-delta columns via `definitions.ts`

**What:** Extend the partner-level table's `ColumnDef[]` with four virtual columns derived from `BatchCurve.projection`. They are NOT in `COLUMN_CONFIGS` (no Snowflake column), they are NOT sorted/filtered as data columns (initially — keep simple), and they read their value from a `Map<batchName, { modeled6: number|null; delta6: number|null; modeled12: ...; delta12: ... }>` passed via `meta`.

**When to use:** Partner-level table only (CONTEXT scope lock). Hidden at root (root has its own `buildRootColumnDefs()` and shouldn't be touched per CONTEXT) and batch-level (account columns from `accountColumnDefs`, not relevant).

**Where:** `src/lib/columns/definitions.ts` — append after the data-column build, gate insertion on the `meta.baselineMode === 'modeled'` flag at consumption time. The cleanest pattern is to build the four virtual columns unconditionally in `buildColumnDefs()` and rely on **column visibility** to hide them when `baselineMode === 'rolling'`. TanStack Table's `columnVisibility` state already handles per-column hide/show.

**Example:**
```typescript
// Source: extends src/lib/columns/definitions.ts (after the COLUMN_CONFIGS map)
// Pattern: virtual derived columns use sentinel-prefixed ids (already a
// codebase convention — __BATCH_COUNT, __DISPLAY_NAME, __avg__).

// Reuse the existing render path through getCellRenderer for percentage type.
// Δ cell uses a custom polarity-colored renderer (Pattern 5).

const MODELED_HORIZONS: Array<{ month: 6 | 12; metricKey: string }> = [
  { month: 6, metricKey: 'COLLECTION_AFTER_6_MONTH' },
  { month: 12, metricKey: 'COLLECTION_AFTER_12_MONTH' },
];

function buildModeledColumns(): ColumnDef<Record<string, unknown>>[] {
  const cols: ColumnDef<Record<string, unknown>>[] = [];
  for (const { month, metricKey } of MODELED_HORIZONS) {
    cols.push({
      id: `__MODELED_AFTER_${month}_MONTH`,
      header: `Modeled ${month}mo`,
      size: 110,
      enableSorting: true,
      enableColumnFilter: false,
      accessorFn: (row) => {
        // Set by data-display via meta.projectionByBatch — see hooks/data-table.tsx
        return null; // populated via cell renderer reading meta below
      },
      cell: (ctx) => {
        const meta = ctx.table.options.meta as TableDrillMeta | undefined;
        const batchName = String(ctx.row.original.BATCH ?? '');
        const modeled = meta?.projectionByBatch?.get(batchName)?.[`modeled${month}`];
        if (modeled == null) return null; // table-body emits em-dash
        return getCellRenderer('percentage', `__MODELED_AFTER_${month}_MONTH`, modeled / 100);
        // Note: formatPercentage multiplies by 100 internally; CurvePoint.recoveryRate
        // is already on the 0..100 scale (see Pitfall 5), so divide by 100 first.
      },
      meta: { type: 'percentage' },
    });
    cols.push({
      id: `__DELTA_VS_MODELED_${month}_MONTH`,
      header: `Δ vs Modeled ${month}mo`,
      size: 130,
      enableSorting: true,
      enableColumnFilter: false,
      accessorFn: () => null,
      cell: (ctx) => {
        const meta = ctx.table.options.meta as TableDrillMeta | undefined;
        const batchName = String(ctx.row.original.BATCH ?? '');
        const delta = meta?.projectionByBatch?.get(batchName)?.[`delta${month}`];
        if (delta == null) return null;
        return createElement(ModeledDeltaCell, { deltaPercent: delta, metricKey });
      },
      meta: { type: 'percentage' },
    });
  }
  return cols;
}

// In buildColumnDefs():
return [anomalyStatusColumn, ...dataColumns, ...buildModeledColumns()];
```

**Visibility gating:** at `data-table.tsx`, after computing `columns`, branch the modeled-column visibility on `baselineMode`. Either pre-set `columnVisibility` defaults or honor the user's manual toggle once they exist (the column picker will show them; rolling mode defaults them hidden).

### Pattern 5: Polarity-colored signed Δ cell

**What:** A small cell formatter that takes `deltaPercent` (signed, where positive = actual exceeds modeled) and `metricKey`, looks up polarity via `getPolarity(metricKey)`, applies green/red text (matching `trend-indicator.tsx`'s convention), and formats the value with sign + percentage.

**Where:** New file `src/components/table/modeled-delta-cell.tsx`. Mirrors `trend-indicator.tsx` but trimmed to a single signed-percentage cell with no tooltip wrapper (the cell IS the value; tooltips on every row would be noisy).

**Existing precedent (`trend-indicator.tsx:30-90`):**
```tsx
const polarity = getPolarity(trend.metric);
const isPositive =
  (trend.direction === 'up' && polarity === 'higher_is_better') ||
  (trend.direction === 'down' && polarity === 'lower_is_better');

let colorClass: string;
if (trend.direction === 'flat') {
  colorClass = 'text-muted-foreground';
} else if (isPositive) {
  colorClass = 'text-success-fg';
} else {
  colorClass = 'text-error-fg';
}
```

**Reusable extraction (recommended):** pull the polarity → color logic into a tiny helper `pickPolarityColor(direction, metricKey)` shared by `TrendIndicator` and the new `ModeledDeltaCell`. Single source of truth for green/red.

**Example:**
```tsx

  metricKey: string;
}

const FLAT_THRESHOLD = 0.5; // percentage points; deltas within ±0.5pp render as flat (gray)

export function ModeledDeltaCell({ deltaPercent, metricKey }: ModeledDeltaCellProps) {
  const polarity = getPolarity(metricKey);
  const direction: 'up' | 'down' | 'flat' =
    deltaPercent > FLAT_THRESHOLD ? 'up' : deltaPercent < -FLAT_THRESHOLD ? 'down' : 'flat';

  const isPositive =
    (direction === 'up' && polarity === 'higher_is_better') ||
    (direction === 'down' && polarity === 'lower_is_better');

  let colorClass: string;
  if (direction === 'flat') {
    colorClass = 'text-muted-foreground';
  } else if (isPositive) {
    colorClass = 'text-success-fg';
  } else {
    colorClass = 'text-error-fg';
  }

  const sign = deltaPercent > 0 ? '+' : '';
  const formatted = `${sign}${deltaPercent.toFixed(1)}pp`;

  // text-body-numeric per AGENTS.md type-token rule for table data cells.
  return <span className={`text-body-numeric ${colorClass}`}>{formatted}</span>;
}
```

**Format note:** "pp" (percentage points) is the unambiguous unit — actual recovery rate − modeled recovery rate is a difference of two percentages, which is a count of percentage points (NOT a relative percentage change). If the planner prefers "%" the cell label can stay; just be aware the math is "pp" semantically.

### Pattern 6: BaselineSelector unification — threading `baselineMode` to chart + table

**What:** `data-display.tsx` already owns `baselineMode` and threads it into `KpiSummaryCards`. Phase 40.1 extends the same wiring to (a) `CollectionCurveChart` (so it can use modeled-mode for the visibility gate; CONTEXT decision: chart projection visibility is mode-gated too — the gate from PRJ-06 only applies when `baselineMode === 'modeled'`) and (b) `DataTable` (via `meta.baselineMode` + `meta.projectionByBatch`).

**Critical CONTEXT lock to call out:** "Switching to 'Modeled curve' reveals modeled + delta columns AND switches KPI deltas AND enables the chart projection-on-narrowed-batch behavior described above. Switching back to 'Rolling avg' reverses all three." So the chart's projection visibility predicate is:

```
showProjection = (baselineMode === 'modeled') && (drillLevel === 'batch' || visibleBatchKeys.length === 1)
```

Not just `drillLevel === 'batch' || visibleBatchKeys.length === 1`. The first clause means in `'rolling'` mode the projection is fully hidden everywhere (return to pre-Phase-40 chart behavior).

**Wiring:**
```tsx
// Source: extends data-display.tsx
// Build a per-batch projection lookup once, reuse for chart + table.
const projectionByBatch = useMemo(() => {
  const map = new Map<string, { modeled6: number|null; delta6: number|null; modeled12: number|null; delta12: number|null }>();
  if (!partnerStats?.curves) return map;
  for (const curve of partnerStats.curves) {
    const m6 = modeledRateAtMonth(curve, 6);
    const m12 = modeledRateAtMonth(curve, 12);
    const a6 = curve.points.find((p) => p.month === 6)?.recoveryRate ?? null;
    const a12 = curve.points.find((p) => p.month === 12)?.recoveryRate ?? null;
    map.set(curve.batchName, {
      modeled6: m6,
      modeled12: m12,
      delta6: m6 != null && a6 != null ? a6 - m6 : null,
      delta12: m12 != null && a12 != null ? a12 - m12 : null,
    });
  }
  return map;
}, [partnerStats?.curves]);
```

Then pass `baselineMode` and `projectionByBatch` into the `DataTable` component via a new prop, and into `CollectionCurveChart` as `baselineMode` + `drillLevel`.

### Anti-Patterns to Avoid

- **Synthesizing an aggregate modeled column at root** — CONTEXT explicit lock. Modeled cols are per-batch only. The root summary rows (one per pair) and partner-aggregate rollups must NOT show them even in modeled mode.
- **Adding modeled columns to `COLUMN_CONFIGS`** — they are not Snowflake columns. Adding them would trip `schema-validator.ts`, pollute the column-allow-list, and falsely advertise warehouse-backed columns. Use sentinel-prefixed virtual `ColumnDef`s.
- **Reading `BatchCurve.projection` directly inside cell renderers via a context** — `meta` is the channel. Adding a new context provider would diverge from the established TanStack Table pattern (`norms`, `trending`, `crossPartnerData` all flow via meta).
- **Tooltip on every Δ cell** — visual noise; the cell IS the answer. The KPI side uses tooltips for explanation but every-row tooltips on a table column would be excessive. Polarity color + sign is enough.
- **Forgetting the AGENTS.md type-token rules** — every new label/cell MUST use the 6 named tokens. Δ cells = `text-body-numeric`; the absent-coverage caption = `text-caption`. NO `text-xs`, `text-sm`, `font-medium`, `font-semibold` outside the allowlist (`src/components/ui/**`, `src/app/tokens/**`, `src/components/tokens/**`).
- **Reset-effect bypass** — when `setBaselineMode` is now persisting, the reset-to-rolling effect at `data-display.tsx:298-306` must continue to call the persisting setter (it will, since the call site uses `setBaselineMode` from the destructured tuple — just keep the same identifier).
- **CSV export coupling** — see Pitfall 6 below. The cell-renderer-only approach for the Δ cell means `row.getValue(col.id)` returns null in CSV export. Fix by adding `accessorFn` that returns the raw delta number; the CSV builder reads `row.getValue` then formats via `meta.type` — `percentage`-type formatters apply. Test before shipping.

## Common Pitfalls

### Pitfall 1: Chart not currently aware of `drillState.level`

**What goes wrong:** PRJ-06 requires the chart to gate projection rendering on `drillState.level === 'batch' || visibleBatchCount === 1`. But `CollectionCurveChart` today receives `curves`, `chartSnapshotRef`, `chartLoadRef`, `chartTypeSelector`, `presetMenu`, `pair`, `rawRows` — no drill level.

**Why it happens:** Phase 40 didn't need it; the chart rendered projection for every batch with coverage.

**How to avoid:**
- Add a `drillLevel` prop to `CollectionCurveChart` (and to `ChartPanel` if it sits between).
- Thread from `data-display.tsx` — `<CollectionCurveChart drillLevel={drillState.level} ... />`.
- Or, alternatively, accept `drillState` as a prop. Prop is fine; no need for a context just for one read.
- The `<ChartPanel>` mount at `data-display.tsx:1133, 1147` is partner-level vs batch-level — easy to thread.

**Warning signs:** "Visibility gate behaves wrong at batch level" — symptom of `drillLevel` not being threaded; the gate falls back to the visibleBatchCount check, which at batch-level is 1 so the gate accidentally works, but the semantics are accidental, not intentional.

### Pitfall 2: `visibleBatchKeys.length === 1` collides with default `defaultVisibleKeys` first-paint

**What goes wrong:** On first paint of a multi-batch partner, `defaultVisibleKeys` is `[batch_0..batch_min(N,8)-1]`; `visibleBatchKeys` filters by `hiddenBatches`. On initial render with a single-batch partner (the partner-aggregate single-batch case is rare but possible), `visibleBatchKeys.length` is 1 and the gate fires — projection renders. That's the desired behavior. But for a single-batch partner that doesn't have modeled coverage, the gate fires AND the coverage-absent caption shows. Confirm the caption logic is consistent with this case.

**Why it happens:** The visibility gate doesn't distinguish "user narrowed legend to one" from "partner only has one batch." Both should render projection (or the absent-coverage caption) by CONTEXT logic — confirmed correct.

**How to avoid:**
- Treat both cases identically (CONTEXT does — "single batch visible" applies regardless of how it became single).
- Document this in a JSDoc on the gate so future readers see the intent.

**Warning signs:** Single-batch partner shows projection (correct); reviewer asks "wait, the user didn't narrow anything" — answer: by CONTEXT, single-batch view = single batch visible = projection on.

### Pitfall 3: First-seen-wins `lenderByBatch` map masks cross-lender batch-name collisions

**What goes wrong:** `use-partner-stats.ts:55-65` walks `partnerRows` once and stores the FIRST-seen `LENDER_ID` for each `batchName`. If two batches in the same partner-product pair share a batch name but belong to different lenders (cross-lender collision), the second is misrouted to the first's `LENDER_ID`, missing its projection or — worse — picking up the wrong projection.

**Why it happens:** Snowflake `BATCH` is a string column with no enforced lender uniqueness. CONTEXT calls this out as Pitfall 1 to investigate.

**How to avoid:**
- **Audit task** (CONTEXT decision: "audit Pitfall 1 paths"): query the live data — for any `(PARTNER_NAME, ACCOUNT_TYPE, BATCH)` triple, count distinct `LENDER_ID`. If always 1, no fix needed; document the audit and ship.
- **Defensive fix** (if collisions exist): change `lenderByBatch` to a `Map<batchName, Set<lenderId>>` or key the map by `(batchName, lenderId)` and require an exact-pair lookup. Either way, expose a way for cells to know they're being asked about an ambiguous batch.
- **Multi-lender partners (Pitfall 1b — Imprint family):** `bounce_imprint`, `bounce_imprint_dp`, `bounce_imprint_dsc` are different `PARTNER_NAME` values, NOT one partner with multiple lenders. So per-partner-product the LENDER_ID should be uniform — but verify by reading partnerRows for the Imprint drill in the audit task.

**Warning signs:** A specific batch's modeled curve looks wrong (numbers that don't match the actuals' shape); a batch shown in the chart but absent from `/api/curves-results` — could be a coverage gap (correct null behavior) or a key-collision bug (silent misroute).

**Recommendation:** Make the audit task a Wave 0 investigation that gates the table-integration work. If audit shows zero collisions, ship as-is. If collisions exist, the fix is small (`Map<batchName, lenderId>` → `Map<\`${batchName}||${lenderId}\`, ...>` and adjust callers).

### Pitfall 4: CSV export must capture modeled values

**What goes wrong:** The CSV builder at `lib/export/csv.ts:93-115` reads `row.getValue(col.id)` to obtain the raw value, then formats via `meta.type`. If the modeled and Δ columns use ONLY `cell:` (no `accessorFn`), `getValue` returns `undefined` and the CSV cell is blank. CONTEXT explicitly flags this as "verify in the plan."

**Why it happens:** `accessorKey` reads from `row.original[key]`; without one, TanStack returns `undefined`. The current `cell:` reads from `meta.projectionByBatch` which is not visible to `row.getValue`.

**How to avoid:**
- **Use `accessorFn`** that reads from `meta.projectionByBatch` — but `accessorFn` does NOT receive `table` or `meta`, only `row.original` and `index`. So the lookup must be embedded in the row.
- **Cleanest fix**: when building `tableData` in `data-display.tsx` at partner-level, stamp each row with `__MODELED_AFTER_6_MONTH`, `__DELTA_VS_MODELED_6_MONTH`, etc. as raw numbers. Then `accessorKey: '__MODELED_AFTER_6_MONTH'` reads them, the cell renderer formats them, and the CSV export reads them. This matches the `__BATCH_COUNT` and `__DISPLAY_NAME` precedent in `root-columns.ts:153-165`.
- **Implication:** the per-batch projection lookup happens during `tableData` shaping (in `data-display.tsx`), not via `meta`. `meta` then only carries the polarity-color metric key (or even that goes via column.meta).

**Warning signs:** Export CSV shows blank cells where modeled values should be; user opens in Excel and sees nothing.

**Recommendation:** Stamp rows in `data-display.tsx` via a new `useMemo` that takes `tableData` + `partnerStats.curves` and returns rows with `__MODELED_AFTER_*` and `__DELTA_VS_MODELED_*` numeric fields. Cells read via standard `accessorKey`; CSV export works automatically. **This is the recommended approach.**

### Pitfall 5: Unit confusion — recoveryRate is 0..100 but formatPercentage expects 0..1

**What goes wrong:** `BatchCurve.points[].recoveryRate` and `BatchCurve.projection[].recoveryRate` are stored on the **0..100 percentage scale** (per `reshape-curves.ts:44` comment + Phase 40 verification). But `formatPercentage()` at `lib/formatting/numbers.ts:49` multiplies its input by 100 (assumes 0..1). Naive `formatPercentage(modeled)` would render 4200% instead of 42%.

**Why it happens:** Most other percentage columns in the app are stored as 0..1 fractions from Snowflake; the curve points were normalized to 0..100 long ago.

**How to avoid:**
- **Convert at the boundary** — pass `modeled / 100` into `formatPercentage` (already shown in Pattern 4).
- **OR** write a custom formatter for the modeled cell that takes 0..100 directly: `${modeled.toFixed(1)}%`. Simpler, matches the existing recovery-rate KPI formatter at `formatPercentage(value, 1)` where `value` is already the rate (need to re-check). On inspection: `formatPercentage` always × 100 — so KPI cards pass values divided by 100 implicitly via their compute path. For consistency, **divide by 100 before formatting** in the modeled cell.
- **For the Δ cell**: deltaPercent is `actual - modeled` where both are 0..100 — so `delta` is in percentage points (pp). The cell shows e.g. `+2.3pp` or `+2.3%`. Per Pattern 5 above, prefer "pp" for clarity.

**Warning signs:** Modeled cell shows 4200% instead of 42%; Δ cell shows +5300% instead of +5.3%.

### Pitfall 6: Type-token compliance for new captions and cells

**What goes wrong:** AGENTS.md mandates: in `src/`, app code uses ONLY the 6 named type tokens (`text-display`, `text-heading`, `text-title`, `text-body`, `text-label`, `text-caption`). No `text-xs`, `text-sm`, `text-base`, `text-lg`, `text-xl`, `text-2xl`. Tokens own weight — no `font-semibold`, `font-medium`, `font-bold` paired. Numeric displays use `.text-display-numeric` / `.text-body-numeric` / `.text-label-numeric`.

**Why it happens:** Easy to default to `text-sm` or pair with `font-medium` when wiring quickly.

**How to avoid:**
- **Modeled cell** — numeric → use `.text-body-numeric` (or no class — table cells inherit the table-body's `.text-body-numeric` per `docs/TYPE-MIGRATION.md:67`). Confirm at `src/components/table/table-body.tsx:65` whether the column type drives the numeric class automatically — if yes, no class needed on the cell wrapper; the formatted span gets it.
- **Δ cell** — numeric, signed → `.text-body-numeric` with the green/red color class. Mirror `trend-indicator.tsx`.
- **Coverage-absent caption** — `text-caption text-muted-foreground` (matches `kpi-summary-cards.tsx:460`).
- **Run `npm run check:tokens`** before each commit. The script greps `src/` for `text-(xs|sm|base|lg|xl|2xl)` and font-weight pairings outside the allowlist.

**Warning signs:** `check:tokens` fails; reviewer asks "why does this cell look different from other numeric cells."

### Pitfall 7: `BaselineSelector` stays partner-only — but rolling-mode gate must still apply at batch-level chart

**What goes wrong:** The `BaselineSelector` UI is only rendered at `drillState.level === 'partner'` (`data-display.tsx:1101`). At batch-level drill, the user sees the chart but no selector — yet the chart's projection visibility is now gated on `baselineMode === 'modeled'` (CONTEXT unification rule). If the user lands at batch-level with `baselineMode === 'rolling'` persisted, projection won't render even though batch-level was the original always-on context. Regression risk.

**Why it happens:** CONTEXT unifies the mode across chart + KPI + table — but the selector is partner-level UI. Batch-level users have no way to flip the mode.

**How to avoid (TWO options — planner picks):**
1. **Render `BaselineSelector` at batch-level too** — small UI addition. The selector already handles `modeledAvailable` gracefully; at batch-level it can show with `modeledAvailable` derived from the single batch's projection coverage.
2. **Decouple the chart projection-visibility gate from `baselineMode` at batch-level** — keep batch-level "always show projection if coverage exists" (Phase 40 behavior) and only apply `baselineMode` gating at partner-level. This may better match user mental model: drilling INTO a batch implies you want to see its projection.

**Recommendation: Option 2.** Drilling into a batch is an intentional act to inspect that batch — projection should always show when coverage exists, regardless of panel-level baseline mode. The mode-gated visibility only matters when the user is looking at multiple batches (partner-aggregate). This also means the `BaselineSelector` can stay partner-level only; no new UI surface needed.

**Concrete predicate (REVISED — supersedes Pattern 6 example):**
```
showProjection = drillLevel === 'batch'
  || (drillLevel === 'partner' && baselineMode === 'modeled' && visibleBatchKeys.length === 1)
```

**Warning signs:** User drills into a batch and sees no projection because `baselineMode === 'rolling'` is persisted; regression of Phase 40 behavior. Surface this for explicit user confirmation in the plan-check stage if Option 2 isn't taken.

**Open Q for planner**: this is a design decision — flag it and recommend Option 2 unless user pushes back.

### Pitfall 8: localStorage-persisted state vs. reset-to-rolling effect

**What goes wrong:** `data-display.tsx:298-306` calls `setBaselineMode('rolling')` when `modeledAvailable` flips false. With the new persisting setter, this WRITES `'rolling'` to localStorage — which is correct. But on the next session for a partner WITH coverage, the user expects their previous 'modeled' selection to be remembered. The reset effect would have overwritten it.

**Why it happens:** The reset effect fires whenever `modeledAvailable` is false — including on initial mount before `partnerStats` resolves and during partner-switching scope changes.

**How to avoid:**
- **Two-state distinction**: distinguish "user-intended mode" (persisted) from "effective mode" (used for rendering). Persist user intent ONLY when the user explicitly clicks the selector. The reset effect changes effective mode without persisting.
- **Implementation**: split into `[storedMode, persistMode] = useBaselineMode()` (the hook) and `[effectiveMode, setEffectiveMode] = useState(storedMode)`. `BaselineSelector.onChange` calls `persistMode(next)` (which also updates effective via the hook). The reset effect calls `setEffectiveMode('rolling')` only.
- **Simpler alternative**: skip the reset effect entirely. If the user's persisted mode is 'modeled' but the current scope has no coverage, every card renders the baseline-absent caption with the "Switch to rolling avg" inline action. The user can recover with one click. This matches the existing per-card recovery-action precedent (`kpi-summary-cards.tsx:462-470`) and AVOIDS the persistence-stomping problem.
- **Recommendation**: simpler alternative — drop the reset effect. The per-card recovery action already handles the empty case. **Document this trade in the plan.**

**Warning signs:** User flips to 'modeled' on Partner A, navigates to Partner B (no coverage), is auto-reset, navigates back to Partner A — expects 'modeled' but gets 'rolling'.

### Pitfall 9: Modeled curve at month X doesn't always exist

**What goes wrong:** `modeledRateAtMonth(curve, 6)` returns `null` when the projection has no point at month 6 (rare but possible — e.g., a batch with sparse projection rows or coverage truncated). The Δ cell would then render `null - actual`, throwing `NaN`.

**Why it happens:** Phase 40 SQL `WHERE PROJECTED_FRACTIONAL IS NOT NULL` drops null months. Most months are present, but edge cases exist.

**How to avoid:**
- Handle `null` explicitly in the projection lookup (`modeledRateAtMonth` already returns null).
- In `projectionByBatch` map, set `delta = null` when modeled or actual is null.
- In the cell renderer, `if (delta == null) return null` → table-body emits em-dash.
- For CSV export, null → blank string (existing CSV builder behavior at `csv.ts:97-99`).

**Warning signs:** `NaN%` in cells; tests against batches with sparse projections fail.

### Pitfall 10: Δ cell formatting precision

**What goes wrong:** Recovery rates are typically 0.x% to 50.x%; deltas are typically -10.x to +10.x percentage points. Showing 0 decimals (`+5pp`) loses too much; 2 decimals (`+5.34pp`) is excessive table-noise. CONTEXT doesn't specify.

**How to avoid:**
- **Recommendation**: 1 decimal (`+5.3pp`). Matches `formatPercentage`'s `decimals: 1` for >10% and `2` for <10% — but for deltas, 1 decimal across the board is consistent.
- For values where `Math.abs(delta) < 0.1`: render `≈0` or "<0.1pp" to avoid showing `+0.0pp` for non-zero values.
- Let the planner make the call but recommend 1 decimal.

**Warning signs:** Cells look noisy (too many decimals) or imprecise (zero decimals masking small movements).

## Open Questions

1. **Should the reset-to-rolling effect at `data-display.tsx:298-306` be removed?**
   - What we know: Phase 40 ships it for safety on scope-change; localStorage persistence makes it stomp persisted intent.
   - What's unclear: whether the per-card "Switch to rolling avg" inline action is sufficient recovery (i.e., the user can fix the stuck state in one click).
   - Recommendation: REMOVE the reset effect. The per-card recovery action handles partial-coverage scopes; full-coverage-absent scopes still surface clearly via the (always-rendered) selector's disabled state and per-card captions. **Surface this for explicit user confirmation in the plan stage.**

2. **`BaselineSelector` rendering at batch-level drill — needed?**
   - What we know: Today only rendered at `drillState.level === 'partner'`. Per Pitfall 7 Recommendation, batch-level chart projection visibility is decoupled from `baselineMode` (always-on when coverage exists).
   - What's unclear: whether the user wants any way to toggle from batch-level (e.g., for the table or future modeled features).
   - Recommendation: keep selector partner-level only. Document the decoupling. If user wants a global selector later, easy follow-up.

3. **Δ cell unit: "pp" vs "%"?**
   - What we know: Mathematically a difference of two percentages is in percentage points (pp). UI conventions split.
   - What's unclear: user preference. CONTEXT doesn't specify.
   - Recommendation: "pp" (precise, unambiguous). If usability testing shows confusion, easy to flip to "%".

4. **Modeled column count — are 4 columns at partner-level too dense?**
   - What we know: CONTEXT § Resolutions #5 — Claude's call, decision was add Modeled + Δ pair per existing milestone column (4 new cols at partner-level partnered with the existing 6mo + 12mo columns).
   - What's unclear: empirical density on real data. CONTEXT explicitly mitigates: gated by mode toggle, default-state table unchanged.
   - Recommendation: ship as decided. If feedback says too dense, easy follow-up to collapse to a single grouped cell.

5. **Coverage-absent caption mount — DataPanel header vs. chart body?**
   - What we know: Phase 40 KPI precedent puts caption beneath the StatCard; chart has DataPanel header (title + actions) and chart body.
   - What's unclear: best location for visual scan.
   - Recommendation: render in DataPanel `actions` slot (right side, where the metric-toggle and segment-toggle live), styled as `text-caption text-muted-foreground`. Keeps the chart body uncluttered and matches the existing chart UI's "metadata in actions" pattern.

6. **Sort behavior on modeled + Δ columns?**
   - What we know: TanStack Table supports sorting on any accessorKey. Default `enableSorting: true` works.
   - What's unclear: should null modeled values sort to top or bottom?
   - Recommendation: default TanStack behavior (nulls to bottom on asc, top on desc). No custom sort fn needed.

7. **Cross-lender batch-name collision — audit-only or defensive fix?**
   - What we know: Pitfall 3. The first-seen-wins map is a real risk in theory; CONTEXT decision is to investigate.
   - What's unclear: whether collisions exist in production data.
   - Recommendation: Wave 0 audit task with a decision gate. If collisions found, defensive fix (Map keyed by `batchName||lenderId`). If not, document the audit and ship.
