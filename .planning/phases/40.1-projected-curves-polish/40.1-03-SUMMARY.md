## Accomplishments

- **Four virtual ColumnDefs appended (PRJ-11).** `src/lib/columns/definitions.ts` defines `MODELED_HORIZONS = [6, 12] as const` and `buildModeledColumns()` helper. The helper flat-maps each horizon into a `{ id, accessorKey, header, cell, meta }` pair: a `Modeled Xmo` percentage column (divides by 100 before `getCellRenderer('percentage', ...)` — Pitfall 5) and a `Δ vs Modeled Xmo` cell rendered via `createElement(ModeledDeltaCell, { deltaPercent, metricKey })` (no division — Plan 01 contract). The four columns are appended to `[anomalyStatusColumn, ...dataColumns]` so they sit at the right edge of the partner-level table after the existing data columns. `COLUMN_CONFIGS` / `ALLOWED_COLUMNS` in `config.ts` are UNCHANGED — these columns have no Snowflake mapping; `schema-validator.ts` boundary preserved.
- **Row-stamping in data-display.tsx (PRJ-11 / RESEARCH Pitfall 4 lock).** New `enrichedTableData` `useMemo` maps each partner-level row to `{ ...row, __MODELED_AFTER_6_MONTH, __MODELED_AFTER_12_MONTH, __DELTA_VS_MODELED_6_MONTH, __DELTA_VS_MODELED_12_MONTH }` using `modeledRateAtMonth` for the modeled values and `curve.points.find((p) => p.month === H)?.recoveryRate` for the actual rates. Δ values stay on the 0..100 scale to match `ModeledDeltaCell`'s contract; modeled values are stamped on 0..100 and divided inside the cell renderer. Root + batch scopes pass through unchanged (`drillState.level !== 'partner'` short-circuits to the original `tableData` reference). The map keying on `batchName` alone is provably safe per the cross-lender collision audit landed in Plan 01 (40.1-01-AUDIT.md, 0 collisions across 477 (pair, batch) entries).
- **CSV export "Just Works" via accessorKey (PRJ-11 / Deferred Idea § "Verify in the plan").** Each new ColumnDef sets BOTH `id` and `accessorKey` to the same sentinel name. The existing CSV export at `csv.ts:95` calls `row.getValue(col.id)` on each visible column — the row-stamp puts the raw numeric there, so the CSV emits the value with the column's `meta.type === 'percentage'` formatter (no custom export-path branch needed). The CSV unit-format mismatch on delta cells (UI shows "+5.3pp", CSV shows "5.30%") is documented inline at the column def and accepted as v1 minor inconsistency, with a flag for v4.2 if WYSIWYG is requested.
- **Visibility gating (PRJ-13 — table half).** `data-table.tsx` accepts a new optional `baselineMode?: BaselineMode` prop (default `'rolling'`). A new visibility effect mirroring the FLT-03 `hidePartnerColumn` pattern at lines 166-179 uses a `lastAppliedBaselineRef` to fire once per `baselineMode` change, calling `columnManagement.setColumnVisibility((prev) => ({ ...prev, ...flip }))` to set all four `__MODELED_* / __DELTA_*` ids to `visible` when `baselineMode === 'modeled'` and `false` otherwise. User's column-picker toggle wins after first apply (last-applied ref guard).
- **DataTable does NOT remount on baselineMode toggle (success criterion lock).** The `key` prop at data-display.tsx:1571 still encodes only drill scope (`level + partner + product + batch`). Toggling `baselineMode` mutates TanStack Table's `columnVisibility` state in place — sort order, column sizing, dimension filters, and column order all survive the toggle.
- **Phase 40.1 milestone closes.** With Plan 03 shipped, the BaselineSelector is the unified toggle for: (a) chart projection `<Line>` visibility (gated to batch-level OR modeled-mode-narrowed-to-one at partner-level — Plan 02), (b) KPI delta calculation (modeled vs actual per card — Phase 40), and (c) table modeled+delta column visibility (this plan). Persistence via localStorage at key `'gsd:baselineMode'` (Plan 01 + Plan 02 wiring). Coverage-absent caption on the chart (Plan 02). Cross-lender collision audit gates lookup safety (Plan 01).
