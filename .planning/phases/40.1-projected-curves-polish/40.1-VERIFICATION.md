### Human Verification Required

Per project convention (`.claude/memory/feedback_testing.md` — "Never push CSS changes blind, always verify in browser"), the following items require browser-visible confirmation before marking the phase complete. None of these block code correctness — the implementations are structurally sound — but UI behavior requires eyes.

#### 1. Aggregate chart legibility restored

**Test:** Drill into a partner with 3+ batches visible (default partner-level view). Do NOT narrow the legend.
**Expected:** Zero dashed projection lines appear in the chart body. Chart reads cleanly.
**Why human:** `projectionScopeAllows` predicate correctly guards this in code, but rendering behavior with real Recharts `<Line hide={true}>` elements requires visual confirmation that hidden lines produce zero visible artifacts.

#### 2. Coverage-absent caption placement and triggering

**Test:** Drill into a specific batch (batch-level drill) where that batch has no modeled projection in the static cache data.
**Expected:** "No modeled curve for this batch" appears in the DataPanel actions bar (not in the chart body). Caption does not overlap controls or break layout.
**Why human:** Caption is conditionally rendered based on `showCoverageAbsent`; requires a batch with null projection to trigger. Layout in the actions flex container needs visual check.

#### 3. Modeled + Δ column appearance on BaselineSelector toggle

**Test:** Drill to partner level. Toggle BaselineSelector to "Modeled curve". Toggle back to "Rolling avg".
**Expected:** On "Modeled curve": four new columns appear at right edge (Modeled 6mo, Δ vs Modeled 6mo, Modeled 12mo, Δ vs Modeled 12mo). On "Rolling avg": all four columns disappear. No full table remount (sort order, column sizing survive the toggle).
**Why human:** Visibility effect wiring is confirmed but actual column appearance and the no-remount guarantee require browser observation.

#### 4. Cell formatting with real data values

**Test:** In modeled mode at partner level with a partner that has projection coverage, inspect the Modeled 6mo and Δ vs Modeled 6mo cells.
**Expected:** Modeled cells show percentage format (e.g., "42.3%"). Δ cells show signed pp format with correct color (e.g., "+5.3pp" green for a collection metric; "-1.7pp" red).
**Why human:** The `/100` divide for percentage cells and the polarity-color logic are correct in code, but visual confirmation with real recovery rate values is needed.

#### 5. localStorage persistence across page reload

**Test:** Toggle to "Modeled curve". Reload the page. Check BaselineSelector state and `localStorage.getItem('gsd:baselineMode')` in dev tools.
**Expected:** BaselineSelector starts on "Modeled curve" after reload. `localStorage.getItem('gsd:baselineMode')` returns `"modeled"`.
**Why human:** Hook implementation is correct but end-to-end persistence requires browser test.

#### 6. CSV export carries new columns

**Test:** In modeled mode at partner level, click Export CSV. Open the downloaded file.
**Expected:** CSV contains the four new columns with numeric values. In rolling mode, CSV does not contain them. Column picker hide/show is reflected in CSV.
**Why human:** The `accessorKey` + row-stamping architecture is the correct path for CSV export (RESEARCH Pitfall 4), but actual CSV output requires manual download and inspection. Note: delta values will show as percentage (e.g., "5.30%") rather than "pp" — this is the documented v1 unit mismatch, not a bug.

## Gaps Summary

Two blocker gaps surfaced during browser UAT (2026-04-26). All automated must-haves verified; the gaps are observable correctness/readability defects in surfaces the static-analysis verifier could not reach.

### Gap 1 — Footer aggregate unit mismatch (blocker)

The 4 new modeled+delta columns display footer averages ~100× too large (modeled) and with wrong unit (`%` on delta cells where the body cells show `pp`). Root cause is in the frontmatter `gaps:` block above. Fix is surgical (per-column footer override on the new col defs); estimated 1 plan, ~2-3 tasks.

### Gap 2 — Header truncation illegible (blocker)

Long column labels truncate to one line and require tooltip hover to read — user reports broadly unusable. Surfaces on the new modeled+delta col labels but reflects a broader complaint about the POL-06 (Phase 38) truncate-with-tooltip pattern. Recommended fix is global (swap truncate → line-clamp-2 in draggable-header.tsx); estimated 1 plan, ~1-2 tasks. Should ship together with Gap 1.

PRJ-09..13 are not yet registered in `v4.1-REQUIREMENTS.md`'s traceability table (executor tooling reported `requirements mark-complete` returned not_found). Bookkeeping — recommend manually adding the five entries during the gap-closure plan or as a separate task.
