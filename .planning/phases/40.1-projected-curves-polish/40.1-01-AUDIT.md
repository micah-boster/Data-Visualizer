# Phase 40.1-01 — Cross-lender batch-name collision audit

**Date:** 2026-04-26
**Plan:** 40.1-01 (foundation primitives)
**Gates:** Plan 02 chart visibility scoping (PRJ-09) + Plan 03 table integration (PRJ-11)
**Audit script:** `scripts/audit-batch-name-collisions.mjs` (deleted post-run; not shipped)

---

## Question

Inside a single `(PARTNER_NAME, ACCOUNT_TYPE)` pair, can a single `BATCH` name map to more than one `LENDER_ID`?

The `lenderByBatch` map at `src/hooks/use-partner-stats.ts:55-65` is **first-seen-wins** — `if (!lenderByBatch.has(batch)) lenderByBatch.set(batch, lender)`. If real production data ever has a `batch → {lender_a, lender_b}` collision inside a single pair, the projection lookup at `${lenderId}||${batchName}` would silently route some batches to the wrong lender's modeled curve (or to nothing at all).

CONTEXT § "Investigate: batch-level coverage gaps" → Pitfall 2 of RESEARCH explicitly directs us to verify this before Plan 03 builds a `projectionByBatch` lookup keyed on `batchName` alone (where the collision risk is unmitigated by the lender prefix).

## Method

1. Read `src/lib/static-cache/batch-summary.json` (the production snapshot embedded in the static-mode fallback).
2. Group rows by `pairKey = ${PARTNER_NAME}||${ACCOUNT_TYPE}`.
3. Within each pair, build `Map<BATCH, Set<LENDER_ID>>`.
4. Flag any entry where the set has size > 1.
5. Spot-check the multi-lender Imprint family explicitly (Pitfall 1b) — `bounce_imprint`, `bounce_imprint_dsc`, `bounce_imprint_dp` all coexist under `Imprint||THIRD_PARTY`.

## Results

| Check | Result |
| --- | --- |
| Cross-lender collisions found (any pair) | **0** |
| Imprint family (`bounce_imprint*`) — distinct lenders observed under `Imprint||THIRD_PARTY` | `bounce_imprint`, `bounce_imprint_dsc`, `bounce_imprint_dp` (3 lenders, 32 batches) |
| Imprint family — any batch name appearing under more than one lender | **0** |

The Imprint family is the worst-case pattern — three distinct lender IDs sharing one `(PARTNER_NAME, ACCOUNT_TYPE)` pair. Each batch's `BATCH` value is unique to its lender (e.g. `IMP_APR_26_DSC_01` is exclusively under `bounce_imprint_dsc`, never under `bounce_imprint`). The naming convention used by Imprint embeds the lender into the batch suffix (`_DSC`, `_DP`), which structurally prevents collisions.

**Collisions found: 0.**

## Decision

**Ship as-is — no defensive fix needed.**

- `src/hooks/use-partner-stats.ts:55-65` retains its current `Map<string, string>` form.
- A single-line comment is added at `:54` referencing this audit so future readers see the empirical result on file.
- Plan 03's planned `projectionByBatch` lookup keyed on `batchName` alone is therefore **safe** for the current data model. If a future Snowflake schema change relaxes the implicit "batch names are lender-unique within a pair" invariant, this audit would need to be re-run (and the defensive `Map<string, Set<string>>` form applied at that point).
