  <action>
    1. Create `src/types/data.ts` -- shared types:
       ```typescript
       export interface DataResponse {
         data: Record<string, unknown>[];
         meta: {
           rowCount: number;
           fetchedAt: string;  // ISO timestamp
           columns: string[];  // Column names in response
         };
         schemaWarnings?: {
           missing: string[];    // Expected columns not found in Snowflake
           unexpected: string[]; // Snowflake columns not in our config
         };
       }
       ```

    2. Create `src/lib/snowflake/types.ts` -- Snowflake-specific types for internal use (connection config interface, query result types).

    3. Create `src/lib/snowflake/connection.ts` -- Connection pool singleton following 01-RESEARCH.md Pattern 1 exactly. Uses `snowflake.createPool()` with `clientSessionKeepAlive: true`. Pool config: `max: 5, min: 0, evictionRunIntervalMillis: 60000, idleTimeoutMillis: 120000`. All credentials from `process.env.SNOWFLAKE_*`. Add a startup check that throws a clear error if any required env var is missing.

    4. Create `src/lib/snowflake/queries.ts` -- Promise wrapper following 01-RESEARCH.md Pattern 2. Export `executeQuery<T>(sqlText, binds?)` that uses `pool.use()` and wraps the callback-based `connection.execute()` in a Promise. Add a timeout of 45 seconds per query to leave headroom under Vercel's 60s limit.

    5. Create `src/lib/columns/config.ts` -- Hardcoded column configuration. Since we don't know the exact 61 column names yet, create a starter config with the columns mentioned in research (PARTNER_NAME, BATCH_NAME, TOTAL_BALANCE, RECOVERY_RATE, ACCOUNT_COUNT) plus placeholder structure. Include:
       - `ColumnConfig` interface: `{ key, label, type, defaultVisible, nullDisplay }`
       - `type` options: `'text' | 'currency' | 'percentage' | 'count' | 'date' | 'number'`
       - `nullDisplay` always `'\u2014'` (em dash)
       - Export `COLUMN_CONFIGS`, `DEFAULT_COLUMNS` (defaultVisible=true ones), `ALLOWED_COLUMNS` (Set of all keys)
       - Add a comment: "Column list will be expanded after first Snowflake connection reveals full schema"

    6. Create `src/lib/columns/schema-validator.ts` -- Following 01-RESEARCH.md schema validation pattern. Queries `INFORMATION_SCHEMA.COLUMNS` for the table, compares against `COLUMN_CONFIGS`, returns `{ valid, missing, unexpected }`. This runs on first data fetch to detect schema drift.

    7. Create `src/app/api/data/route.ts` -- GET route handler following 01-RESEARCH.md Pattern 4:
       - `export const dynamic = 'force-dynamic'` to prevent caching
       - Accept `columns` query param (comma-separated column names)
       - Validate requested columns against `ALLOWED_COLUMNS` -- reject unknown column names (SQL injection prevention)
       - If no columns param or `*`, use `DEFAULT_COLUMNS`
       - Build SQL: `SELECT {validated_columns} FROM agg_batch_performance_summary`
       - Run schema validation on first request (cache result in module scope)
       - Return `DataResponse` shape with data, meta (rowCount, fetchedAt, columns), and schemaWarnings if any
       - On error: return `{ error: string, details: string }` with status 500
       - Log errors to console with context (query, error message)

    IMPORTANT: Never expose Snowflake credentials in error responses. The `details` field should contain the error message, NOT connection strings or credentials.
  </action>

    `pnpm build` compiles without TypeScript errors. If .env.local exists with valid credentials: `curl http://localhost:3000/api/data` returns JSON with `data` array and `meta.rowCount`. If no credentials: the route returns a 500 with a clear error message about missing configuration (not a stack trace with credentials).
  </verify>
  <done>Snowflake connection pool, query executor, column config, schema validator, and /api/data route all implemented. API returns real Snowflake data when credentials are configured, clear error when not.</done>
