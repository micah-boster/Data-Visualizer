### Scenario A — Legacy view round-trips cleanly (CHRT-02 happy path)

1. Open DevTools → Application → Local Storage → `http://localhost:3000` → `bounce-dv-saved-views`
2. Replace its value with this hand-seeded legacy payload (one user-created view carrying the pre-Phase-35 shape):

```json
[
  {
    "id": "test-legacy-view-01",
    "name": "Legacy Recovery Test",
    "createdAt": 1700000000000,
    "snapshot": {
      "sorting": [],
      "columnVisibility": {},
      "columnOrder": [],
      "columnFilters": {},
      "dimensionFilters": {},
      "columnSizing": {},
      "chartsExpanded": true,
      "chartState": {
        "metric": "amount",
        "hiddenBatches": [],
        "showAverage": true,
        "showAllBatches": false
      }
    }
  }
]
```

3. Hard reload the page (Cmd+Shift+R)
4. Open the saved-views menu (sidebar or toolbar) → select "Legacy Recovery Test"

**Expected:**
- [ ] View loads without a red-screen error
- [ ] Chart renders with the "Dollars Collected" metric selected (legacy `metric: 'amount'` preserved)
- [ ] DevTools console shows NO uncaught errors
- [ ] DevTools console shows NO `[chartState migration]` warning (this is the happy path — legacy shape has all required fields)
- [ ] Re-open DevTools → Application → Local Storage. The stored `chartState` STILL carries the legacy shape (no `type`/`version` fields). This confirms the CONTEXT-locked "in-memory migration + write-back on next save" decision — nothing was written back yet.

5. Save the view (click Save / rename / any action that triggers persistence). Re-inspect localStorage.

**Expected (self-healing write-back):**
- [ ] `chartState` now carries `"type": "collection-curve"` and `"version": 2` in the persisted JSON. Migration self-heals on next save.

### Scenario B — Malformed legacy seed falls back gracefully

1. Replace localStorage `bounce-dv-saved-views` with:

```json
[
  {
    "id": "test-broken-view",
    "name": "Broken Fixture",
    "createdAt": 1700000000000,
    "snapshot": {
      "sorting": [],
      "columnVisibility": {},
      "columnOrder": [],
      "columnFilters": {},
      "dimensionFilters": {},
      "columnSizing": {},
      "chartState": {
        "hiddenBatches": []
      }
    }
  }
]
```

Note: `chartState.metric` is deliberately missing — required field. This hits the `migrateChartState` "missing required fields" failure branch.

2. Hard reload, open saved-views menu, select "Broken Fixture".

**Expected:**
- [ ] View loads without crashing
- [ ] Chart renders with the DEFAULT_COLLECTION_CURVE preset (Recovery Rate %, showAverage: true, showAllBatches: false)
- [ ] DevTools console shows EXACTLY ONE `[chartState migration] legacy record missing required fields, falling back` warning (or the unknown-shape variant if the legacy-detection branch didn't catch it — either is acceptable)
- [ ] No stack trace / error state / red screen

### Scenario C — 3 default views still load after clearing storage (CHRT-01)

1. DevTools → Application → Local Storage → `bounce-dv-saved-views` → DELETE the key entirely
2. Hard reload the page
3. Open the saved-views menu

**Expected:**
- [ ] All 3 seeded views appear: "Financial Overview", "Outreach Performance", "New Batches"
- [ ] Click each one — they each load and render their respective chart/table states without error
- [ ] "Financial Overview" and "New Batches" both have working chart state (the v2-authored defaults from defaults.ts)
- [ ] "Outreach Performance" has no chart state by design — verify no crash, chart section just uses its own internal default
- [ ] DevTools console is clean (no migration warnings — defaults were authored in v2 shape, migration's idempotent branch passes them through)

### Scenario D — Consumer narrow works (no `as` escape hatch regressed)
