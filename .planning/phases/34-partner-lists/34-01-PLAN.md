  name: z.string().min(1),

src/hooks/use-partner-lists.ts (NEW) — public API contract:
```typescript
export interface UsePartnerListsResult {
  lists: PartnerList[];
  createList: (input: {
    name: string;
    partnerIds: string[];
    filters: PartnerListFilters;
    source: 'attribute' | 'manual';
  }) => PartnerList;          // returns the created list (so callers can activate it immediately)
  deleteList: (id: string) => PartnerList | undefined;   // returns the removed list for undo toast
  restoreList: (list: PartnerList) => void;
  renameList: (id: string, name: string) => void;
  updateList: (
    id: string,
    patch: Partial<Pick<PartnerList, 'name' | 'filters' | 'partnerIds'>>,
  ) => void;                                              // edit-mode bulk updater; merges patch + bumps updatedAt
  refreshList: (
    id: string,
    rows: Array<Record<string, unknown>>,
  ) => { added: number; removed: number } | undefined;   // only runs on source === 'attribute'
  getList: (id: string) => PartnerList | undefined;
  hasListWithName: (name: string) => boolean;
}

export function usePartnerLists(): UsePartnerListsResult;
```

Critical implementation requirements (Pitfall 3 — SSR hydration mismatch):
- `'use client'` directive at top.
- `useState<PartnerList[]>([])` initializer (NOT lazy initializer reading localStorage).
- `hasHydrated = useRef(false)`.
- First `useEffect([])`: call `setLists(loadPartnerLists())`, then `hasHydrated.current = true`. No else branch — an empty array stays empty.
- Second `useEffect([lists])`: `if (!hasHydrated.current) return; persistPartnerLists(lists);`.
- Do NOT seed with defaults — `getDefaultPartnerLists()` returns `[]`, and new users start empty.

Mutation callbacks (all wrapped in `useCallback([])` with no stale-state dependencies — use functional setState):
- `createList({ name, partnerIds, filters, source })` — builds the `PartnerList` with `crypto.randomUUID()` and `Date.now()` for both timestamps. Returns the created list synchronously (constructed value, not `useState` read) so callers can immediately feed the new ID into `setActiveListId`.
- `deleteList(id)` — captures the deleted list via a closure variable inside functional setState, returns it for the undo toast. Mirror the pattern in `use-saved-views.ts:96-106`.
- `restoreList(list)` — `setLists(prev => [...prev, list])`. Append-only restore; acceptable that undo lands the list at the bottom (matches saved-views behavior).
- `renameList(id, name)` — map over list, update `name: name.trim()` and bump `updatedAt`. No-op if trimmed name is empty.
- `updateList(id, patch)` — bulk edit-mode updater used by the edit flow in Plan 03. Accepts `Partial<Pick<PartnerList, 'name' | 'filters' | 'partnerIds'>>`. Functional setState: map over `prev`, for the matching `id` return `{ ...list, ...patch, updatedAt: Date.now() }`. If `patch.name` is supplied, trim it before merging; skip the patch (return prev list unchanged) when the trimmed name is empty. Natural analog to `renameList` / `deleteList` — same shape, just multi-field. Do NOT change `source`; `source` is locked at creation (Pitfall 7). If callers somehow pass `source` in the patch, ignore it at the type level (the `Pick` already excludes it).
- `refreshList(id, rows)` — ONLY operates when `list.source === 'attribute'`; bails with `undefined` return otherwise (Pitfall 7 — hand-picked lists don't get Refresh). For attribute lists: call `evaluateFilters(rows, list.filters)`, build new `partnerIds = Array.from(matching)`, compute diff `{ added, removed }` (counts against previous `list.partnerIds`), update list with new partnerIds + bumped `updatedAt`, return the diff so callers can toast "Refreshed 'X': +3 added, -1 removed" (Pitfall 6).
- `getList(id)`, `hasListWithName(name)` — memoized read helpers; include `lists` in deps.

- Do NOT call `persistPartnerLists` synchronously inside mutation callbacks — let the second `useEffect([lists])` handle persistence to avoid double-writes.
- Do NOT read `localStorage` outside of the two `useEffect` blocks.
